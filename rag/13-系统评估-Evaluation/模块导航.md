# 09-系统评估-Evaluation 学习导航

## 📋 模块概述

系统评估是 RAG 系统的**质量保证**环节。如果说构建 RAG 系统是"造车"，那么系统评估就是"验车"——确保系统真正达到预期效果。没有评估，我们无法知道优化是否有效，也无法发现系统的薄弱环节。

**核心思想**：
- **不能度量，就无法改进**：评估提供量化指标
- **多维度评估**：检索质量、生成质量、端到端效果
- **持续监控**：不仅在开发阶段，生产环境也需要评估

**学习目标**：
- 理解 RAG 系统评估的重要性和评估维度
- 掌握 RAGAS 评估框架的使用
- 学会使用 TruLens 进行可观测性监控
- 了解 DeepEval 和 LlamaIndex 评估工具
- 能够设计和实施完整的评估方案

**难度级别**：⭐⭐⭐ 中等

**前置知识**：
- 完成 [00-简单RAG-SimpleRAG](../00-简单RAG-SimpleRAG/模块导航.md) 模块 ⭐ 重要

- 理解 RAG 系统的完整流程
- 了解评估指标的基本概念（准确率、召回率等）

**关键技术**：
- RAGAS - RAG 专用评估框架
- TruLens - LLM 应用可观测性平台
- DeepEval - 深度评估工具
- LlamaIndex Evaluation - LlamaIndex 内置评估模块
- Faithfulness - 忠实度评估

## 🎯 为什么需要系统评估？

### 核心问题

**1. 如何知道系统是否工作良好？**
```python
# 没有评估
user_query = "孙悟空的技能有哪些？"
answer = rag_system.query(user_query)
# 答案看起来不错，但真的准确吗？相关吗？完整吗？

# 有评估
evaluation_result = {
    "faithfulness": 0.85,      # 答案忠实于检索到的文档
    "answer_relevancy": 0.92,  # 答案与问题高度相关
    "context_relevancy": 0.78  # 检索到的上下文相关
}
# 现在我们知道：答案质量不错，但检索可以优化
```

**2. 如何比较不同的优化方案？**
```python
# 方案A：使用BGE嵌入模型
result_A = {"precision": 0.75, "recall": 0.68}

# 方案B：使用OpenAI嵌入模型
result_B = {"precision": 0.82, "recall": 0.71}

# 结论：方案B更好，提升了7%的准确率
```

**3. 如何发现系统的薄弱环节？**
```python
evaluation = {
    "retrieval_precision": 0.92,  # 检索很好
    "generation_quality": 0.65,   # 生成质量差
    "end_to_end_accuracy": 0.68   # 整体效果受生成影响
}
# 发现：需要优化生成环节（提示词、模型选择）
```

### RAG 系统的评估维度

```
RAG 系统评估
├── 检索质量评估
│   ├── Context Precision（上下文精确度）
│   ├── Context Recall（上下文召回率）
│   └── Context Relevancy（上下文相关性）
├── 生成质量评估
│   ├── Faithfulness（忠实度）
│   ├── Answer Relevancy（答案相关性）
│   └── Answer Correctness（答案正确性）
└── 端到端评估
    ├── Overall Accuracy（整体准确性）
    ├── Response Time（响应时间）
    └── User Satisfaction（用户满意度）
```

### 最小可运行示例：RAGAS 评估

**文件：** `01-RAGAS.py` ⭐ 推荐从这里开始

```python
from datasets import Dataset
from ragas.metrics import Faithfulness, AnswerRelevancy
from ragas.llms import LangchainLLMWrapper
from ragas.embeddings import LangchainEmbeddingsWrapper
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from ragas import evaluate

# 1. 准备评估数据
data = {
    "question": ["孙悟空的主要技能有哪些？"],
    "answer": ["孙悟空的主要技能包括金箍棒攻击、筋斗云移动和七十二变变身。"],
    "contexts": [[
        "孙悟空拥有多种强大技能，包括使用金箍棒进行战斗。",
        "他还可以使用筋斗云快速移动，以及七十二变进行变身。"
    ]]
}
dataset = Dataset.from_dict(data)

# 2. 初始化评估模型
llm = LangchainLLMWrapper(ChatOpenAI(model="gpt-3.5-turbo"))
embeddings = LangchainEmbeddingsWrapper(OpenAIEmbeddings())

# 3. 定义评估指标
metrics = [
    Faithfulness(llm=llm),                          # 忠实度
    AnswerRelevancy(llm=llm, embeddings=embeddings) # 答案相关性
]

# 4. 执行评估
result = evaluate(dataset, metrics)
print(f"忠实度: {result['faithfulness']:.4f}")
print(f"答案相关性: {result['answer_relevancy']:.4f}")
```

**预期输出**：
```
忠实度: 0.9500
答案相关性: 0.9200
```

**核心流程**：
```
准备数据 → 初始化模型 → 定义指标 → 执行评估 → 分析结果
```

## 🔍 核心评估工具详解

### 1. RAGAS - RAG 专用评估框架 ⭐ 推荐

**文件：** `01-RAGAS.py`

**功能**：
RAGAS（Retrieval Augmented Generation Assessment）是专门为 RAG 系统设计的评估框架，提供多个维度的自动化评估指标。

**核心评估指标**：

#### 1.1 Faithfulness（忠实度）
**定义**：评估生成的答案是否忠实于检索到的上下文。

**工作原理**：
```
1. 将答案分解为多个简单陈述
2. 对每个陈述，判断是否可以从上下文中推断出来
3. 计算可推断陈述的比例
```

**示例**：
```python
# 上下文
contexts = ["孙悟空使用金箍棒战斗", "他可以使用筋斗云移动"]

# 答案
answer = "孙悟空使用金箍棒战斗，可以飞行"

# 分解陈述
statements = [
    "孙悟空使用金箍棒战斗",  # ✅ 可从上下文推断
    "孙悟空可以飞行"          # ❌ 上下文中没有提到
]

# 忠实度 = 1/2 = 0.5
```

**核心代码**：
```python
from ragas.metrics import Faithfulness

faithfulness = Faithfulness(llm=llm)
result = evaluate(dataset, [faithfulness])
print(f"忠实度: {result['faithfulness']:.4f}")
```

**适用场景**：
- 检测模型是否产生幻觉（hallucination）
- 确保答案基于事实
- 评估上下文的利用程度

#### 1.2 Answer Relevancy（答案相关性）
**定义**：评估生成的答案与用户问题的相关程度。

**工作原理**：
```
1. 使用嵌入模型将问题和答案向量化
2. 计算问题向量和答案向量的余弦相似度
3. 相似度越高，相关性越好
```

**示例**：
```python
# 问题
question = "孙悟空的主要技能有哪些？"

# 答案A
answer_A = "孙悟空的主要技能包括金箍棒、筋斗云和七十二变"
# 相关性: 0.95 ✅ 高度相关

# 答案B
answer_B = "孙悟空是《西游记》中的角色"
# 相关性: 0.45 ❌ 相关性低
```

**核心代码**：
```python
from ragas.metrics import AnswerRelevancy

relevancy = AnswerRelevancy(llm=llm, embeddings=embeddings)
result = evaluate(dataset, [relevancy])
print(f"答案相关性: {result['answer_relevancy']:.4f}")
```

**适用场景**：
- 评估答案是否回答了用户的问题
- 检测答案是否偏离主题
- 优化生成策略

#### 1.3 Context Relevancy（上下文相关性）
**定义**：评估检索到的上下文与用户问题的相关程度。

**工作原理**：
```
1. 对每个检索到的文档，评估其与问题的相关性
2. 使用 LLM 判断文档是否包含回答问题所需的信息
3. 计算相关文档的比例
```

**适用场景**：
- 评估检索系统的质量
- 优化检索策略
- 减少无关上下文的干扰

#### 1.4 其他指标

**Context Precision（上下文精确度）**：
- 评估检索到的相关文档的排序质量
- 相关文档是否排在前面

**Context Recall（上下文召回率）**：
- 评估是否检索到了所有相关文档
- 需要标注的真实答案（ground truth）

**Answer Correctness（答案正确性）**：
- 评估答案的事实正确性
- 需要标注的真实答案

### 评估指标对比

| 指标 | 评估对象 | 需要 LLM | 需要 Embedding | 需要 Ground Truth |
|------|---------|---------|---------------|------------------|
| Faithfulness | 答案 vs 上下文 | ✅ | ❌ | ❌ |
| Answer Relevancy | 答案 vs 问题 | ✅ | ✅ | ❌ |
| Context Relevancy | 上下文 vs 问题 | ✅ | ❌ | ❌ |
| Context Precision | 上下文排序 | ✅ | ❌ | ✅ |
| Context Recall | 上下文完整性 | ✅ | ❌ | ✅ |
| Answer Correctness | 答案正确性 | ✅ | ✅ | ✅ |

### 2. TruLens - LLM 应用可观测性平台 ⭐ 推荐

**文件：** `02-Trulens.py`

**功能**：
TruLens 是一个用于 LLM 应用的可观测性和评估平台，可以跟踪、调试和评估 RAG 系统的每个环节。

**核心特性**：

#### 2.1 应用跟踪（Application Tracing）
使用 `@instrument` 装饰器标记需要跟踪的函数：

```python
from trulens.apps.app import instrument

class RAG:
    @instrument
    def retrieve(self, query: str):
        """检索相关文档"""
        results = vector_store.query(query)
        return results
    
    @instrument
    def generate(self, query: str, context: list):
        """生成答案"""
        answer = llm.invoke(f"Context: {context}\nQuestion: {query}")
        return answer
    
    @instrument
    def query(self, query: str):
        """完整的RAG流程"""
        context = self.retrieve(query)
        return self.generate(query, context)
```

**效果**：
- 记录每个函数的输入、输出
- 记录执行时间
- 记录调用链路

#### 2.2 评估反馈（Feedback Functions）
定义评估维度和评估器：

```python
from trulens.core import Feedback, Select
from trulens.providers.openai import OpenAI as TruLensOpenAI

provider = TruLensOpenAI(model_engine="gpt-4")

# 1. Groundedness（内容真实性）
f_groundedness = Feedback(
    provider.groundedness_measure_with_cot_reasons,
    name="Groundedness"
).on(Select.RecordCalls.retrieve.rets).on_output()

# 2. Answer Relevance（答案相关性）
f_answer_relevance = Feedback(
    provider.relevance_with_cot_reasons,
    name="Answer Relevance"
).on_input().on_output()

# 3. Context Relevance（上下文相关性）
f_context_relevance = Feedback(
    provider.context_relevance_with_cot_reasons,
    name="Context Relevance"
).on_input().on(Select.RecordCalls.retrieve.rets[:]).aggregate(np.mean)
```

**选择器（Selector）说明**：
- `.on_input()`：评估应用的输入（用户查询）
- `.on_output()`：评估应用的输出（生成的答案）
- `.on(Select.RecordCalls.retrieve.rets)`：评估 retrieve 函数的返回值

#### 2.3 可观测性仪表板
```python
from trulens.core import TruSession

session = TruSession()

# 查看评估结果
leaderboard = session.get_leaderboard()
print(leaderboard)
```

**输出示例**：
```
App Name | Version | Groundedness | Answer Relevance | Context Relevance
---------|---------|--------------|------------------|------------------
RAG      | base    | 0.85         | 0.92             | 0.78
RAG      | v2      | 0.90         | 0.94             | 0.82
```

**优势**：
- ✅ 实时监控：在生产环境中持续监控
- ✅ 详细跟踪：记录完整的调用链路
- ✅ 版本对比：比较不同版本的性能
- ✅ 可视化：提供 Web 界面查看结果

**适用场景**：
- 开发阶段的调试和优化
- 生产环境的性能监控
- A/B 测试和版本对比
- 问题诊断和根因分析

### 常见问题

**Q1: 应该使用哪个评估工具？**
A: 根据需求选择：
- **RAGAS**：专注 RAG 评估，指标全面，推荐首选
- **TruLens**：需要可观测性和实时监控
- **DeepEval**：需要多维度评估和报告
- **LlamaIndex**：使用 LlamaIndex 框架

**Q2: 评估需要多少数据？**
A: 
- **最少**：10-20 个测试用例（快速验证）
- **推荐**：50-100 个测试用例（可靠评估）
- **理想**：200+ 个测试用例（全面评估）

**Q3: 如何准备评估数据？**
A: 
1. **收集真实用户查询**：从日志中提取
2. **人工标注答案**：标注正确答案（ground truth）
3. **多样化覆盖**：覆盖不同类型的查询
4. **包含边界情况**：包含困难和模糊的查询

**Q4: 评估指标多少分算好？**
A: 
- **Faithfulness**: > 0.8 良好，> 0.9 优秀
- **Answer Relevancy**: > 0.85 良好，> 0.92 优秀
- **Context Relevancy**: > 0.75 良好，> 0.85 优秀

**Q5: 评估成本高吗？**
A: 
- **RAGAS**：每次评估调用 LLM，成本中等
- **TruLens**：持续监控，成本较高
- **优化方法**：
  - 使用开源模型（如 DeepSeek）降低成本
  - 采样评估（不是每次都评估）
  - 缓存评估结果

**Q6: 如何持续评估？**
A: 
1. **开发阶段**：每次优化后评估
2. **测试阶段**：全面评估测试集
3. **生产阶段**：
   - 采样评估（如 1% 的请求）
   - 定期批量评估（如每周）
   - 监控关键指标（如响应时间）

### 避坑指南

**坑1：过度依赖单一指标**
- ❌ 只看 Answer Relevancy
- ✅ 综合多个指标（Faithfulness + Relevancy + Context）

**坑2：测试数据不够多样化**
- ❌ 只测试简单查询
- ✅ 包含各种类型：简单、复杂、模糊、边界情况

**坑3：忽略成本**
- ❌ 每次请求都评估
- ✅ 采样评估或定期批量评估

**坑4：没有 Ground Truth**
- ❌ 无法评估 Context Recall 和 Answer Correctness
- ✅ 至少标注部分数据作为基准

**坑5：评估与实际脱节**
- ❌ 评估数据与真实用户查询差异大
- ✅ 使用真实用户查询进行评估
