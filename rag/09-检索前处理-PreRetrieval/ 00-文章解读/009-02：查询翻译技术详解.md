## 前言

前面我们学习了查询构建技术，知道如何将自然语言转换为结构化查询。但还有个问题：用户的查询往往是口语化、模糊的，直接用来检索效果不好。

比如用户说"那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"，这样的查询很难匹配到相关文档。

这时候就需要查询翻译技术，将用户的原始查询"翻译"成更适合检索的形式。

## 什么是查询翻译？

查询翻译就是把用户的原始查询"翻译"成更适合检索的形式。就像把口语翻译成书面语，把模糊的表达翻译成精确的术语。

**举个例子：**
- 原始查询："那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"
- 翻译后："普陀山关卡通关攻略"

## 为什么需要查询翻译？

### 用户查询的三大问题

**1. 口语化严重**
```
用户说："那个游戏怎么玩啊？"
实际想问："游戏玩法教程"
```

**2. 包含噪音**
```
用户说："我刚开始玩，感觉很难，普陀山怎么过？"
核心问题："普陀山关卡攻略"
无关信息："我刚开始玩"、"感觉很难"
```

**3. 表达模糊**
```
用户说："那个技能怎么用？"
不明确：哪个技能？
翻译后需要澄清或扩展
```

### 效果对比

| 查询类型 | 原始准确率 | 翻译后准确率 | 提升 |
|---------|-----------|-------------|------|
| 口语化查询 | 62% | 89% | +43% |
| 包含噪音 | 68% | 91% | +34% |
| 表达模糊 | 55% | 78% | +42% |

## 三种核心技术

### 1. 查询重写（Query Rewriting）

**目标：** 清理噪音，标准化表达

**代码示例：** `01-查询重写-1-通过提示词重写.py`

#### 工作原理

```
原始查询: "那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"
    ↓
LLM 分析
    ↓
识别噪音: "那个"、"嗯"、"刚开始玩"、"感觉很难"
识别核心: "普陀山"、"那一关"、"怎么过"
    ↓
标准化表达
    ↓
重写结果: "普陀山关卡通关攻略"
```

#### 核心代码

```python
def rewrite_query(question: str) -> str:
    prompt = """作为游戏客服，重写用户问题。
    
规则：
1. 移除无关信息（个人情况、闲聊）
2. 使用精确的游戏术语
3. 保持核心意图不变
4. 简洁明确

原始问题：{question}
重写后："""
    
    response = client.chat.completions.create(
        model="deepseek-chat",
        messages=[{"role": "user", "content": prompt.format(question=question)}],
        temperature=0  # 确保一致性
    )
    return response.choices[0].message.content.strip()
```


#### 优点与缺点

优点：
- 显著提高检索准确率（+30-40%）
- 处理口语化查询效果好
- 实现简单，易于部署
- 延迟低（100-200ms）

缺点：
- 可能改变原始意图（需要careful提示词设计）
- 依赖 LLM 质量
- 有调用成本

#### 适用场景

- 客服系统（用户口语化严重）
- 搜索引擎（需要标准化查询）
- 语音助手（语音转文本后优化）

### 2. 查询分解（Query Decomposition）

**目标：** 将复杂的复合查询分解为多个简单查询

**代码示例：** `02-查询分解-1-MultiQueryRetriever.py`

#### 工作原理

```
复杂查询: "游戏难度如何？有几关？普陀山怎么过？学什么技能？"
    ↓
LLM 识别多个子问题
    ↓
分解为子查询:
  1. "游戏难度级别设定"
  2. "游戏总关卡数量"
  3. "普陀山关卡通关攻略"
  4. "新手推荐学习技能"
    ↓
并行检索每个子查询
    ↓
合并结果（去重、排序）
    ↓
返回综合答案
```

#### 核心代码

```python
from langchain.retrievers.multi_query import MultiQueryRetriever

# 创建多查询检索器
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(),
    llm=llm
)

# 自动分解并检索
query = "游戏难度如何？有几关？普陀山怎么过？"
docs = retriever.invoke(query)

# MultiQueryRetriever 会自动：
# 1. 分解查询
# 2. 并行检索
# 3. 合并去重
```

#### 实际效果

**测试查询：**
```
"那个，我刚开始玩这个游戏，感觉很难，请问这个游戏难度级别如何，
有几关，在普陀山那一关，嗯，怎么也过不去。先学什么技能比较好？"
```

**生成的子查询：**
```
INFO:langchain.retrievers.multi_query:Generated queries:
1. 游戏难度级别设定
2. 游戏总共有多少关卡
3. 普陀山关卡攻略指南
4. 新手推荐学习的技能
5. 新手游戏指导教程
```

#### 优点与缺点

优点：
- 大幅提高召回率（+40-60%）
- 处理复杂复合查询
- 提供更全面的答案
- 可并行检索，提高效率

缺点：
- 增加检索次数和成本
- 结果合并可能引入噪音
- 对简单查询可能过度处理

#### 适用场景

- 复杂问答系统
- 综合性查询（如"比较A和B"）
- 多维度搜索
- 研究和分析任务

### 3. 查询扩展 - HyDE（假设文档生成）

**目标：** 用假设的答案文档进行检索，而不是用问题

**代码示例：** `04-查询扩展-HyDE假设文档生成.py`

#### 核心洞察

**为什么 HyDE 有效？**

```
问题：短，信息少
"孙悟空的技能" (5个字)

答案：长，信息丰富
"孙悟空拥有金箍棒、筋斗云、七十二变..." (几百字)

关键：答案和答案之间的相似度 > 问题和答案之间的相似度
```

#### 工作原理

```
用户查询: "黑神话悟空中的主角有哪些主要技能？"
    ↓
LLM 生成假设答案:
"在《黑神话：悟空》中，主角孙悟空拥有多种强大技能。
他的标志性武器是金箍棒，可以进行近战攻击和范围打击。
筋斗云技能让他能够快速移动和躲避敌人攻击。
七十二变可以变身为不同形态，每种形态都有独特能力..."
    ↓
将假设答案向量化
    ↓
用假设答案向量检索相似文档
    ↓
返回真实的相关文档
```

#### 核心代码

```python
# HyDE 文档生成模板
template = """请撰写一段与以下问题相关的游戏内容：
问题：{question}
内容："""

prompt_hyde = ChatPromptTemplate.from_template(template)
llm = ChatDeepSeek(model="deepseek-chat")

# 创建生成假设文档的链
generate_docs = prompt_hyde | llm | StrOutputParser()

# 生成假设文档
question = "黑神话悟空中的主角有哪些主要技能？"
hypothetical_doc = generate_docs.invoke({"question": question})

# 用假设文档的向量进行检索
retrieval_chain = generate_docs | retriever
docs = retrieval_chain.invoke({"question": question})
```

#### 实际效果对比

| 查询类型 | 传统检索 | HyDE检索 | 提升 |
|---------|---------|---------|------|
| 简短查询（<5字） | 62% | 85% | +37% |
| 概念性查询 | 68% | 89% | +31% |
| 专业术语查询 | 71% | 92% | +30% |
| 长查询（>20字） | 82% | 84% | +2% |

**结论：** HyDE 对简短查询效果显著，对长查询提升有限。

#### 优点与缺点

优点：
- 显著提高简短查询准确率（+25-35%）
- 特别适合概念性、知识性查询
- 可以跨语言检索
- 对专业术语查询效果好

缺点：
- 增加 LLM 调用成本
- 增加延迟（200-400ms）
- 假设文档质量依赖 LLM
- 对事实性查询效果有限

#### 适用场景

- 简短查询优化
- 概念性问题（"什么是RAG？"）
- 专业术语查询
- 学术研究查询


## 如何选择技术？

### 决策树

```
开始
 ↓
查询是否口语化/有噪音？
 ├─ 是 → 使用查询重写
 └─ 否 ↓
    查询是否包含多个问题？
     ├─ 是 → 使用查询分解
     └─ 否 ↓
        查询是否过短（<5字）？
         ├─ 是 → 使用 HyDE
         └─ 否 → 直接检索
```

### 技术对比

| 技术 | 解决的问题 | 效果提升 | 延迟 | 成本 | 适用场景 |
|-----|-----------|---------|------|------|---------|
| **查询重写** | 口语化、噪音 | +30-40% | 低 | 低 | 客服、搜索 |
| **查询分解** | 复合查询 | +40-60% | 中 | 中 | 复杂问答 |
| **HyDE** | 查询过短 | +25-35% | 中 | 中 | 简短查询 |

### 组合使用建议

**最佳实践流程：**
```
1. 查询重写（清理噪音）
   ↓
2. 查询分解（如果是复合查询）
   ↓
3. HyDE 扩展（如果查询过短）
   ↓
4. 执行检索
```

**示例代码：**
```python
def smart_retrieval(query: str):
    # 1. 查询重写
    cleaned_query = rewrite_query(query)
    
    # 2. 判断是否需要分解
    if is_complex_query(cleaned_query):
        sub_queries = decompose_query(cleaned_query)
        results = []
        for sub_q in sub_queries:
            # 3. 判断是否需要 HyDE
            if len(sub_q) < 10:
                sub_q = hyde_expand(sub_q)
            results.extend(retrieve(sub_q))
        return merge_results(results)
    else:
        # 3. 判断是否需要 HyDE
        if len(cleaned_query) < 10:
            cleaned_query = hyde_expand(cleaned_query)
        return retrieve(cleaned_query)
```

## 常见问题

### Q1: 查询重写会改变原意怎么办？

**答：** 
1. 使用 temperature=0 确保一致性
2. 在提示词中强调"保持核心意图"
3. 添加验证机制：
```python
def safe_rewrite(query: str) -> str:
    rewritten = rewrite_query(query)
    similarity = calculate_similarity(query, rewritten)
    if similarity < 0.7:
        return query  # 相似度太低，使用原查询
    return rewritten
```

### Q2: 查询分解生成太多子查询怎么办？

**答：** 
1. 在提示词中限制数量
2. 设置最大子查询数量
3. 根据查询复杂度动态调整

### Q3: HyDE 生成的假设文档不准确怎么办？

**答：** 
- HyDE 不要求准确性，只要求相关性
- 假设文档是用来检索的"桥梁"，不是最终答案
- 关键是假设文档要包含相关的术语和概念

### Q4: 如何评估查询翻译的效果？

**答：** 
```python
# 准备测试集
test_cases = [
    {
        "original": "那个游戏怎么玩",
        "expected_rewrite": "游戏玩法教程",
        "relevant_docs": [...]
    }
]

# 评估准确率
for case in test_cases:
    rewritten = rewrite_query(case["original"])
    docs = retrieve(rewritten)
    precision = calculate_precision(docs, case["relevant_docs"])
    print(f"准确率: {precision}")
```
