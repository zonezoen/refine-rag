# 05-检索前处理-PreRetrieval 学习导航

## 📋 模块概述

检索前处理是 RAG 系统的**智能优化层**，它在用户查询和向量检索之间架起一座桥梁。如果说向量检索是"找答案"，那么检索前处理就是"问对问题"。通过优化查询本身，我们可以显著提升检索的准确性和覆盖面。

**核心思想**：用户的原始查询往往不是最适合检索的形式，通过智能处理可以：
- 将模糊的查询转换为精确的表达
- 将复杂的查询分解为多个子查询
- 将查询路由到最合适的数据源
- 生成假设文档来改善检索效果

**学习目标**：
- 理解查询优化对检索效果的重要性
- 掌握查询重写、分解、扩展的技术
- 学会使用 HyDE 技术提升检索准确性
- 掌握查询路由的实现方法
- 理解 Text-to-SQL 和 Text-to-Cypher 的应用

**难度级别**：⭐⭐⭐ 中等

**前置知识**：
- 完成 [00-简单RAG-SimpleRAG](../00-简单RAG-SimpleRAG/模块导航.md) 模块

- 理解向量检索的基本原理
- 了解提示词工程的基础知识

**关键技术**：
- Query Rewriting - 查询重写
- Query Decomposition - 查询分解
- Query Expansion - 查询扩展（HyDE）
- Query Routing - 查询路由

## 🎯 为什么需要检索前处理？

### 核心问题

用户的原始查询往往存在以下问题：

**1. 表达不精确**
```python
# 用户查询
"那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"

# 问题：
- 包含口语化表达："那个"、"嗯"
- 包含无关信息："刚开始玩"、"感觉很难"
- 关键信息模糊："那一关"

# 优化后的查询
"普陀山关卡攻略"
```

**2. 查询过于复杂**
```python
# 用户查询
"游戏难度如何？有几关？普陀山怎么过？学什么技能？"

# 问题：包含4个不同的问题，单次检索难以覆盖

# 分解后的子查询
queries = [
    "游戏难度级别设定",
    "游戏总关卡数量",
    "普陀山关卡攻略",
    "新手推荐技能"
]
```

**3. 查询过于简单**
```python
# 用户查询
"悟空技能"

# 问题：过于简单，检索结果可能不够精确

# 扩展后的查询（HyDE）
"孙悟空在游戏中拥有多种强大的技能，包括金箍棒攻击、筋斗云移动、
七十二变变身等。这些技能可以通过升级和修炼来增强..."
```

### 效果对比

| 场景 | 原始查询 | 优化后 | 效果提升 |
|------|---------|--------|---------|
| 口语化查询 | "那个游戏怎么玩" | "游戏玩法教程" | +40% |
| 复杂查询 | "难度？关卡？攻略？" | 分解为3个子查询 | +60% |
| 简单查询 | "悟空" | HyDE生成详细描述 | +35% |
| 多数据源 | "Python列表用法" | 路由到python_docs | +50% |

### 最小可运行示例：查询重写

**文件：** `02-查询翻译/01-查询重写-1-通过提示词重写.py`

```python
from openai import OpenAI
from os import getenv

# 初始化DeepSeek客户端
client = OpenAI(
    base_url="https://api.deepseek.com",
    api_key=getenv("DEEPSEEK_API_KEY")
)

def rewrite_query(question: str) -> str:
    """使用LLM重写用户查询"""
    prompt = """作为一个游戏客服人员，你需要帮助用户重写他们的问题。

规则：
1. 移除无关信息（如个人情况、闲聊内容）
2. 使用精确的游戏术语表达
3. 保持问题的核心意图
4. 将模糊的问题转换为具体的查询

原始问题：{question}
请直接给出重写后的查询。"""

    response = client.chat.completions.create(
        model="deepseek-chat",
        messages=[{"role": "user", "content": prompt.format(question=question)}],
        temperature=0
    )
    return response.choices[0].message.content.strip()

# 测试
query = "那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去。先学什么技能比较好？"
print(f"原始查询：{query}")
print(f"重写查询：{rewrite_query(query)}")
```

**预期输出**：
```
原始查询：那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去。先学什么技能比较好？
重写查询：普陀山关卡攻略，新手推荐技能
```

**核心流程**：
```
用户查询 → LLM分析 → 移除噪音 → 提取关键信息 → 标准化表达 → 优化后的查询
```

### 1. 查询构建（Query Construction）

**目录：** `01-查询构建/`

**核心内容**：
将自然语言查询转换为结构化查询语言（SQL、Cypher、元数据过滤器）。

#### 1.1 Text-to-SQL - 自然语言转SQL

**目录：** `Text2SQL/`

**功能**：将用户的自然语言问题转换为SQL查询语句。

**核心代码**：
```python
from langchain_community.utilities import SQLDatabase
from langchain.chains import create_sql_query_chain
from langchain_deepseek import ChatDeepSeek

# 连接数据库
db = SQLDatabase.from_uri("sqlite:///sakila.db")

# 创建Text-to-SQL链
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
chain = create_sql_query_chain(llm, db)

# 自然语言查询
question = "有多少个演员？"
sql_query = chain.invoke({"question": question})
print(f"生成的SQL: {sql_query}")

# 执行查询
result = db.run(sql_query)
print(f"查询结果: {result}")
```

**应用场景**：
- 数据库问答系统
- 商业智能报表
- 数据分析助手
- 企业知识库查询

**优点**：
- ✅ 精确查询：SQL查询比向量检索更精确
- ✅ 结构化数据：适合处理表格数据
- ✅ 复杂查询：支持聚合、连接等复杂操作

**缺点**：
- ❌ 需要数据库：依赖结构化数据库
- ❌ 学习成本：需要理解数据库结构
- ❌ 错误处理：生成的SQL可能有语法错误

#### 1.2 Text-to-Cypher - 自然语言转图查询

**目录：** `Text2Cypher/`

**功能**：将自然语言转换为Cypher查询语言（用于图数据库）。

**核心代码**：
```python
from langchain_community.graphs import Neo4jGraph
from langchain.chains import GraphCypherQAChain
from langchain_deepseek import ChatDeepSeek

# 连接Neo4j图数据库
graph = Neo4jGraph(
    url="bolt://localhost:7687",
    username="neo4j",
    password="password"
)

# 创建Text-to-Cypher链
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
chain = GraphCypherQAChain.from_llm(llm, graph=graph)

# 自然语言查询
question = "孙悟空的师傅是谁？"
result = chain.invoke({"query": question})
print(f"答案: {result['result']}")
```

**应用场景**：
- 知识图谱问答
- 社交网络分析
- 推荐系统
- 关系挖掘

#### 1.3 元数据过滤器构建

**目录：** `构建元数据Filter/`

**功能**：从查询中提取元数据过滤条件，缩小检索范围。

**核心代码**：
```python
from langchain.chains.query_constructor.base import AttributeInfo
from langchain.retrievers.self_query.base import SelfQueryRetriever
from langchain_deepseek import ChatDeepSeek

# 定义元数据字段
metadata_field_info = [
    AttributeInfo(
        name="author",
        description="视频作者",
        type="string"
    ),
    AttributeInfo(
        name="publish_year",

# 创建自查询检索器
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)

```

**优势**：
- 自动提取过滤条件
- 减少检索范围

#### 2.4 查询扩展 - HyDE（Hypothetical Document Embeddings）

**文件：** `04-查询扩展-HyDE假设文档生成.py` ⭐ 推荐学习

**功能**：
生成假设性的答案文档，用这个文档的向量进行检索，而不是直接用查询的向量。

**核心原理**：
```
用户查询 → LLM生成假设答案 → 将假设答案向量化 → 用假设答案向量检索 → 找到真实答案
```

**为什么有效？**
- 查询通常很短，向量表示不够丰富
- 假设答案更详细，向量表示更准确
- 答案和答案之间的相似度 > 问题和答案之间的相似度

**核心代码**：
```python
from langchain.prompts import ChatPromptTemplate
from langchain_deepseek import ChatDeepSeek

# HyDE文档生成模板
template = """请撰写一段与以下问题相关的游戏内容：
问题：{question}
内容："""

prompt = ChatPromptTemplate.from_template(template)
llm = ChatDeepSeek(model="deepseek-chat")

# 生成假设文档
generate_docs = prompt | llm | StrOutputParser()
question = "黑神话悟空中的主角有哪些主要技能？"
hypothetical_doc = generate_docs.invoke({"question": question})

# 用假设文档的向量进行检索
retrieval_chain = generate_docs | retriever
docs = retrieval_chain.invoke({"question": question})
```

**示例**：
```python
# 查询
"孙悟空的主要技能"

# 生成的假设文档
"孙悟空在游戏中拥有多种强大的技能。他的标志性武器是金箍棒，
可以进行近战攻击和范围打击。筋斗云技能让他能够快速移动和躲避敌人。
七十二变可以变身为不同的形态，每种形态都有独特的能力..."

# 用这个假设文档的向量去检索，效果更好！
```

**优势**：
- ✅ 提高检索准确性：答案向量比问题向量更准确
- ✅ 处理简短查询：扩展查询的语义信息
- ✅ 跨语言检索：可以生成不同语言的假设文档

**适用场景**：
- 简短查询优化
- 跨语言检索
- 专业术语查询
- 概念性问题

### 3. 查询路由（Query Routing）

**目录：** `03-查询路由/`

**核心内容**：
根据查询内容，智能地将查询路由到最合适的数据源或处理流程。

#### 3.1 逻辑路由（Logical Routing）

**文件：** `01-逻辑路由.py` ⭐ 推荐学习

**功能**：
基于规则和LLM判断，将查询路由到预定义的数据源。

**核心代码**：
```python
from typing import Literal
from langchain_core.pydantic_v1 import BaseModel, Field
from langchain_deepseek import ChatDeepSeek

# 定义路由模型
class RouteQuery(BaseModel):
    datasource: Literal["python_docs", "js_docs", "golang_docs"] = Field(
        description="根据用户问题，选择最适合的数据源"
    )

# 创建路由器
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
structured_llm = llm.with_structured_output(RouteQuery)

# 路由提示
system = """你是将用户问题路由到合适数据源的专家。
- python_docs: Python语言相关问题
- js_docs: JavaScript语言相关问题
- golang_docs: Go语言相关问题"""

prompt = ChatPromptTemplate.from_messages([
    ("system", system),
    ("human", "{question}"),
])

router = prompt | structured_llm

# 执行路由
question = "Python中的列表和元组有什么区别？"
result = router.invoke({"question": question})
print(f"路由到: {result.datasource}")  # 输出: python_docs
```

**应用场景**：
- 多数据源问答系统
- 技术文档查询
- 智能客服分流
- 微服务路由

**优势**：
- ✅ 提高检索效率：只在相关数据源中搜索
- ✅ 提高准确性：专门的数据源提供更精确的答案
- ✅ 可扩展：易于添加新的数据源

#### 3.2 语义路由（Semantic Routing）

**文件：** `02-语义路由.py`

**功能**：
基于语义相似度进行路由，不需要预定义规则。

**核心原理**：
```
查询 → 向量化 → 计算与各路由的相似度 → 选择最相似的路由
```

**应用场景**：
- 动态路由场景
- 无法预定义规则的情况
- 需要灵活路由的系统

### 常见问题

**Q1: 查询重写、分解、扩展，应该用哪个？**
A: 根据场景选择：
- **查询重写**：用户查询口语化、包含噪音 → 使用重写
- **查询分解**：用户查询包含多个问题 → 使用分解
- **查询扩展（HyDE）**：用户查询过于简短 → 使用HyDE
- **组合使用**：先重写，再分解，效果更好

**Q2: HyDE为什么有效？**
A: 
- 问题通常很短："孙悟空技能"
- 答案通常很长："孙悟空拥有金箍棒、筋斗云、七十二变..."
- 答案和答案之间的相似度 > 问题和答案之间的相似度
- 用假设答案的向量检索，更容易找到真实答案

**Q3: 查询路由什么时候用？**
A: 
- 有多个数据源（如：Python文档、JS文档、Go文档）
- 不同类型的查询需要不同的处理流程
- 需要提高检索效率（只在相关数据源中搜索）

**Q4: Text-to-SQL和向量检索有什么区别？**
A: 
- **Text-to-SQL**：
  - 适合结构化数据（表格）
  - 精确查询（聚合、统计、过滤）
  - 需要数据库支持
- **向量检索**：
  - 适合非结构化数据（文本、文档）
  - 语义相似度搜索
  - 不需要数据库

**Q5: 检索前处理会增加延迟吗？**
A: 
- 会增加一些延迟（通常100-500ms）
- 但检索准确性提升显著（20-60%）
- 可以通过缓存、并行处理来优化
- 权衡：延迟 vs 准确性

**Q6: 如何评估检索前处理的效果？**
A: 
1. **检索准确率**：检索到的文档是否相关
2. **检索覆盖率**：是否覆盖了所有相关文档
3. **用户满意度**：用户是否满意答案
4. **A/B测试**：对比有无检索前处理的效果

### 避坑指南

**坑1：过度优化**
- ❌ 不要对所有查询都使用所有技术
- ✅ 根据查询特点选择合适的技术
- ✅ 简单查询可能不需要优化

**坑2：忽略成本**
- ❌ 每次优化都调用LLM，成本高
- ✅ 对常见查询使用缓存
- ✅ 简单规则优先，复杂情况才用LLM

**坑3：丢失原始意图**
- ❌ 查询重写后完全改变了原意
- ✅ 在提示词中强调"保持核心意图"
- ✅ 对重写结果进行验证

**坑4：查询分解过度**
- ❌ 将简单查询分解为过多子查询
- ✅ 只对复合查询进行分解
- ✅ 限制子查询数量（通常3-5个）

**坑5：路由错误**
- ❌ 路由到错误的数据源
- ✅ 提供清晰的数据源描述
- ✅ 使用结构化输出确保路由准确性
- ✅ 添加置信度评估

## 📊 技术对比总结

### 查询优化技术对比

| 技术 | 适用场景 | 效果提升 | 成本 | 难度 |
|------|---------|---------|------|------|
| **查询重写** | 口语化、有噪音 | +30-40% | 低 | ⭐⭐ |
| **查询分解** | 复合查询 | +40-60% | 中 | ⭐⭐⭐ |
| **HyDE** | 简短查询 | +25-35% | 中 | ⭐⭐⭐ |
| **查询路由** | 多数据源 | +40-50% | 低 | ⭐⭐ |
| **Text-to-SQL** | 结构化数据 | +60-80% | 中 | ⭐⭐⭐⭐ |

### 何时使用哪种技术？

**决策树**：
```
用户查询
  ├─ 包含口语化表达？
  │   └─ 是 → 查询重写
  │
  ├─ 包含多个问题？
  │   └─ 是 → 查询分解
  │
  ├─ 查询过于简短？
  │   └─ 是 → HyDE扩展
  │
  ├─ 有多个数据源？
  │   └─ 是 → 查询路由
  │
  └─ 查询结构化数据？
      └─ 是 → Text-to-SQL
```

### 组合使用建议

**最佳实践流程**：
```
1. 查询重写（去噪、标准化）
   ↓
2. 查询路由（选择数据源）
   ↓
3. 查询分解（如果是复合查询）
   ↓
4. HyDE扩展（如果查询过短）
   ↓
5. 执行检索
```

def rewrite_query(query: str) -> str:

## 📈 效果评估

### 评估指标

**1. 检索准确率（Precision）**
```python
precision = 相关文档数 / 检索到的文档总数
```

**2. 检索召回率（Recall）**
```python
recall = 检索到的相关文档数 / 所有相关文档数
```

**3. F1分数**
```python
f1 = 2 * (precision * recall) / (precision + recall)
```

**4. 用户满意度**
- 用户是否满意答案
- 是否需要重新提问
- 平均交互轮次
