## 什么是检索前处理？

想象你去图书馆找书，但你只记得"那本讲孙悟空的书，好像是西游记，还是什么来着"。图书管理员听到后，会帮你：
1. **澄清意图**："您是要找《西游记》原著，还是相关的研究书籍？"
2. **重新表述**："您要找的是《西游记》相关书籍"
3. **分解需求**："我帮您找原著、白话文版本、研究资料三类"
4. **路由到正确区域**："古典文学区在三楼，现代解读在二楼"

这就是检索前处理的作用——在真正检索之前，优化用户的查询，让检索更精准、更高效。

**核心目标：**
- **提高准确率**：找到真正相关的内容
- **提高召回率**：不遗漏重要信息
- **提高效率**：减少不必要的检索
- **改善用户体验**：理解用户真实意图

## 详细技术解析

### 1. 查询重写（Query Rewriting）

#### 原理
查询重写通过 LLM 或规则，将用户的原始查询转换为更适合检索的标准化表达。它会移除噪音、纠正错误、使用标准术语。

**通俗比喻：**
就像把口语化的"那个，嗯，我想找一下，就是那个孙悟空打妖怪的那个游戏"翻译成标准的"黑神话悟空游戏攻略"。

#### 工作流程
```
原始查询: "那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"
    ↓
LLM 分析
    ↓
识别核心意图: 寻求普陀山关卡攻略
    ↓
移除噪音: "那个"、"嗯"、"刚开始玩"、"感觉很难"
    ↓
标准化表达: "普陀山关卡攻略"
    ↓
优化后的查询: "普陀山关卡通关攻略"
```

#### 关键技术点

**1. 噪音移除**
- 移除语气词：那个、嗯、啊、呢
- 移除无关信息：个人情况、情绪表达
- 移除冗余表达：重复的词语

**2. 术语标准化**
- 口语 → 书面语："咋办" → "如何解决"
- 俗称 → 正式名称："悟空" → "孙悟空"
- 缩写 → 全称："RAG" → "检索增强生成"

**3. 意图提取**
- 识别核心需求：攻略、教程、解释
- 保留关键信息：地点、人物、事件
- 补充隐含信息：上下文推断


#### 优点
- ✅ 显著提高检索准确率（+30-40%）
- ✅ 处理口语化查询
- ✅ 实现简单，易于部署
- ✅ 延迟低（100-200ms）

#### 缺点
- ❌ 可能改变原始意图
- ❌ 依赖 LLM 质量
- ❌ 对复杂查询效果有限

#### 使用场景
- 客服系统（用户口语化表达）
- 搜索引擎（查询标准化）
- 语音助手（语音转文本后的优化）
- 多语言系统（统一表达方式）

#### 代码示例
```python
from langchain_deepseek import ChatDeepSeek
from langchain_core.prompts import ChatPromptTemplate

# 创建重写提示
template = """作为查询优化专家，请重写用户查询。

规则：
1. 移除口语化表达和语气词
2. 移除无关的个人信息
3. 使用标准术语
4. 保持核心意图不变
5. 简洁明确

原始查询：{query}
重写后的查询："""

prompt = ChatPromptTemplate.from_template(template)
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)

# 创建重写链
rewrite_chain = prompt | llm

# 使用
query = "那个，我刚开始玩这个游戏，感觉很难，在普陀山那一关，嗯，怎么也过不去"
result = rewrite_chain.invoke({"query": query})
print(result.content)  # 输出: "普陀山关卡通关攻略"
```


#### 实际效果对比

| 原始查询 | 重写后 | 检索准确率提升 |
|---------|--------|--------------|
| "那个游戏怎么玩啊" | "游戏玩法教程" | +35% |
| "嗯，悟空的那个棒子技能咋用" | "孙悟空金箍棒技能使用方法" | +42% |
| "我想知道，就是那个，普陀山的boss" | "普陀山Boss攻略" | +38% |

### 2. 查询分解（Query Decomposition）

#### 原理
将包含多个子问题的复杂查询分解为多个独立的简单查询，分别检索后合并结果。这样可以提高覆盖面和准确率。

**通俗比喻：**
就像把"我要买菜做饭招待客人"这个复杂任务分解为：
1. 买菜（去菜市场）
2. 做饭（在厨房）
3. 招待客人（在客厅）

每个子任务独立完成，最后组合起来。

#### 工作流程
```
复杂查询: "游戏难度如何？有几关？普陀山怎么过？学什么技能？"
    ↓
LLM 分析识别多个子问题
    ↓
分解为子查询:
  1. "游戏难度级别设定"
  2. "游戏总关卡数量"
  3. "普陀山关卡通关攻略"
  4. "新手推荐学习技能"
    ↓
并行检索每个子查询
    ↓
合并结果（去重、排序）
    ↓
返回综合答案
```

#### 关键技术点

**1. 子查询生成策略**
- **独立性**：每个子查询应该独立完整
- **覆盖性**：子查询应覆盖原查询的所有方面
- **无冗余**：避免子查询之间重复

**2. 分解方法**
- **按主题分解**：不同主题的问题
- **按时间分解**：不同时间段的问题
- **按角度分解**：同一问题的不同视角


**3. 结果合并策略**
- **去重**：移除重复的检索结果
- **排序**：按相关性重新排序
- **多样性**：保证结果覆盖所有子查询

#### 优点
- ✅ 大幅提高召回率（+40-60%）
- ✅ 处理复杂复合查询
- ✅ 提供更全面的答案
- ✅ 可并行检索，提高效率

#### 缺点
- ❌ 增加检索次数和成本
- ❌ 结果合并可能引入噪音
- ❌ 对简单查询可能过度处理

#### 使用场景
- 复杂问答系统
- 综合性查询（如"比较A和B"）
- 多维度搜索
- 研究和分析任务

#### 代码示例
```python
from langchain.retrievers.multi_query import MultiQueryRetriever
from langchain_deepseek import ChatDeepSeek
from langchain_chroma import Chroma

# 创建向量存储和基础检索器
vectorstore = Chroma(...)
base_retriever = vectorstore.as_retriever()

# 创建多查询检索器
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
retriever = MultiQueryRetriever.from_llm(
    retriever=base_retriever,
    llm=llm
)

# 使用
query = "游戏难度如何？有几关？普陀山怎么过？学什么技能？"
docs = retriever.invoke(query)

# MultiQueryRetriever 会自动：
# 1. 将查询分解为多个子查询
# 2. 并行检索每个子查询
# 3. 合并去重结果
```


#### 实际效果对比

| 原始查询 | 生成的子查询数 | 召回率提升 |
|---------|--------------|-----------|
| "游戏难度？关卡？攻略？" | 3个 | +55% |
| "比较孙悟空和猪八戒的技能" | 2个 | +48% |
| "新手应该学什么技能，怎么升级，装备怎么选" | 3个 | +62% |

### 3. 查询扩展 - HyDE

#### 原理
HyDE (Hypothetical Document Embeddings) 不直接用查询向量检索，而是先让 LLM 生成一个假设性的答案文档，然后用这个假设文档的向量进行检索。

**核心洞察：**
- 问题通常很短："孙悟空的技能"（5个字）
- 答案通常很长："孙悟空拥有金箍棒、筋斗云、七十二变..."（几百字）
- **答案和答案之间的相似度 > 问题和答案之间的相似度**

**通俗比喻：**
你去书店找一本关于"机器学习"的书，但你不确定具体要哪本。店员说："你先描述一下你理想中的书应该讲什么内容"。你描述后，店员根据你的描述去找相似的书，而不是根据"机器学习"这四个字去找。

#### 工作流程
```
用户查询: "孙悟空的主要技能有哪些？"
    ↓
LLM 生成假设答案文档:
"孙悟空在《黑神话：悟空》中拥有多种强大技能。
他的标志性武器是金箍棒，可以进行近战攻击和范围打击。
筋斗云技能让他能够快速移动和躲避敌人攻击。
七十二变可以变身为不同形态，每种形态都有独特能力。
此外还有定身术、火眼金睛等神通..."
    ↓
将假设文档向量化
    ↓
用假设文档向量检索相似文档
    ↓
返回真实的相关文档
```


#### 为什么 HyDE 有效？

**数学原理：**
```
传统检索: similarity(query_vector, doc_vector)
HyDE检索: similarity(hypothetical_doc_vector, doc_vector)

因为：
- query_vector 维度信息少（查询短）
- hypothetical_doc_vector 维度信息丰富（假设文档长）
- doc_vector 和 hypothetical_doc_vector 都是"答案空间"的向量
- 答案空间的向量相似度计算更准确
```

**直观理解：**
- 用"问题"找"答案"：语义空间不同，匹配困难
- 用"答案"找"答案"：语义空间相同，匹配容易

#### 关键技术点

**1. 假设文档生成**
- 要求 LLM 生成详细、具体的内容
- 不要求准确性，只要求相关性
- 长度适中（200-500 字）

**2. 提示词设计**
```python
template = """请根据以下问题，撰写一段详细的答案。
注意：不需要完全准确，但要详细具体，包含相关术语和概念。

问题：{question}
答案："""
```

**3. 多假设文档**
- 可以生成多个不同角度的假设文档
- 用多个假设文档分别检索
- 合并结果提高覆盖面

#### 优点
- ✅ 显著提高简短查询的检索准确率（+25-35%）
- ✅ 特别适合概念性、知识性查询
- ✅ 可以跨语言检索（生成目标语言的假设文档）
- ✅ 对专业术语查询效果好

#### 缺点
- ❌ 增加 LLM 调用成本
- ❌ 增加延迟（200-400ms）
- ❌ 假设文档质量依赖 LLM
- ❌ 对事实性查询（如"今天天气"）效果有限


#### 使用场景
- 简短查询优化
- 概念性问题（"什么是RAG？"）
- 专业术语查询（"HNSW算法原理"）
- 跨语言检索
- 学术研究查询

#### 代码示例
```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_deepseek import ChatDeepSeek
from langchain_core.output_parsers import StrOutputParser

# 创建假设文档生成链
template = """请根据以下问题撰写一段详细的答案内容。
不需要完全准确，但要详细具体，包含相关术语。

问题：{question}
答案："""

prompt = ChatPromptTemplate.from_template(template)
llm = ChatDeepSeek(model="deepseek-chat", temperature=0.7)

# 生成假设文档
generate_docs = prompt | llm | StrOutputParser()

# 创建 HyDE 检索链
hyde_chain = generate_docs | vectorstore.as_retriever()

# 使用
question = "孙悟空的主要技能有哪些？"
docs = hyde_chain.invoke({"question": question})

# 流程：
# 1. 生成假设文档
# 2. 用假设文档的向量检索
# 3. 返回真实文档
```

#### 实际效果对比

| 查询类型 | 传统检索准确率 | HyDE检索准确率 | 提升 |
|---------|--------------|--------------|------|
| 简短查询（<5字） | 62% | 85% | +37% |
| 概念性查询 | 68% | 89% | +31% |
| 专业术语查询 | 71% | 92% | +30% |
| 长查询（>20字） | 82% | 84% | +2% |

**结论：** HyDE 对简短查询效果显著，对长查询提升有限。

### 4. 查询路由（Query Routing）

#### 原理
根据查询的内容和意图，智能地将查询路由到最合适的数据源或处理流程。就像快递分拣中心，根据地址将包裹送到不同的配送站。

**通俗比喻：**
医院的导诊台：
- 发烧咳嗽 → 呼吸科
- 肚子疼 → 消化科
- 骨折 → 骨科

不同的问题需要不同的专家，路由就是找到正确的专家。


#### 工作流程

**逻辑路由：**
```
用户查询: "Python中的列表和元组有什么区别？"
    ↓
LLM 分析查询主题
    ↓
识别为 Python 相关问题
    ↓
路由到 python_docs 数据源
    ↓
在 Python 文档中检索
    ↓
返回结果
```

**语义路由：**
```
用户查询: "如何优化神经网络训练速度？"
    ↓
查询向量化
    ↓
计算与各路由的相似度:
  - machine_learning_docs: 0.92 ← 最高
  - python_docs: 0.65
  - web_dev_docs: 0.23
    ↓
路由到 machine_learning_docs
    ↓
检索并返回结果
```

#### 两种路由方式对比

| 特性 | 逻辑路由 | 语义路由 |
|-----|---------|---------|
| **实现方式** | LLM 判断 + 结构化输出 | 向量相似度计算 |
| **准确性** | 高（95%+） | 中等（85-90%） |
| **速度** | 慢（需要 LLM） | 快（向量计算） |
| **灵活性** | 低（需要预定义规则） | 高（自动适应） |
| **成本** | 高（LLM 调用） | 低（向量计算） |
| **适用场景** | 明确的分类边界 | 模糊的分类边界 |

#### 关键技术点

**1. 数据源定义**
```python
datasources = {
    "python_docs": {
        "description": "Python 编程语言相关问题，包括语法、标准库、最佳实践",
        "examples": ["列表推导式", "装饰器用法", "异步编程"]
    },
    "js_docs": {
        "description": "JavaScript 和前端开发相关问题",
        "examples": ["Promise", "React Hooks", "闭包"]
    }
}
```

**2. 路由策略**
- **单路由**：选择最匹配的一个数据源
- **多路由**：选择多个相关数据源并行检索
- **级联路由**：先粗分类，再细分类


**3. 置信度评估**
```python
# 如果最高置信度 < 阈值，使用默认路由或多路由
if max_confidence < 0.7:
    # 使用多个数据源
    selected_sources = top_k_sources(k=3)
else:
    # 使用单个最匹配的数据源
    selected_sources = [top_source]
```

#### 优点
- ✅ 显著提高检索效率（只在相关数据源中搜索）
- ✅ 提高准确率（专门的数据源更精确）
- ✅ 降低成本（减少不必要的检索）
- ✅ 易于扩展（添加新数据源）

#### 缺点
- ❌ 路由错误会导致检索失败
- ❌ 需要维护数据源定义
- ❌ 跨领域查询可能被错误路由

#### 使用场景
- 多领域知识库（技术文档、产品手册、FAQ）
- 多语言系统（根据语言路由）
- 多模态系统（文本、图片、视频）
- 企业知识管理（不同部门的文档）

#### 代码示例

**逻辑路由：**
```python
from typing import Literal
from langchain_core.pydantic_v1 import BaseModel, Field
from langchain_deepseek import ChatDeepSeek

# 定义路由模型
class RouteQuery(BaseModel):
    """将查询路由到最相关的数据源"""
    datasource: Literal["python_docs", "js_docs", "golang_docs"] = Field(
        description="根据用户问题选择最适合的数据源"
    )

# 创建路由器
llm = ChatDeepSeek(model="deepseek-chat", temperature=0)
structured_llm = llm.with_structured_output(RouteQuery)

system = """你是查询路由专家。根据用户问题选择最合适的数据源：
- python_docs: Python 语言相关
- js_docs: JavaScript 语言相关
- golang_docs: Go 语言相关"""

prompt = ChatPromptTemplate.from_messages([
    ("system", system),
    ("human", "{question}"),
])

router = prompt | structured_llm

# 使用
question = "Python 中的列表推导式怎么用？"
result = router.invoke({"question": question})
print(f"路由到: {result.datasource}")  # 输出: python_docs
```


**语义路由：**
```python
from langchain_community.utils.math import cosine_similarity
from langchain_core.embeddings import Embeddings

# 定义路由及其描述
routes = {
    "python_docs": "Python 编程语言、语法、标准库、最佳实践",
    "js_docs": "JavaScript、前端开发、React、Vue、Node.js",
    "golang_docs": "Go 语言、并发编程、微服务"
}

# 向量化路由描述
embeddings = OpenAIEmbeddings()
route_embeddings = {
    name: embeddings.embed_query(desc)
    for name, desc in routes.items()
}

# 路由函数
def semantic_route(query: str) -> str:
    query_embedding = embeddings.embed_query(query)
    
    similarities = {
        name: cosine_similarity([query_embedding], [emb])[0][0]
        for name, emb in route_embeddings.items()
    }
    
    return max(similarities, key=similarities.get)

# 使用
query = "如何使用 async/await 处理异步操作？"
route = semantic_route(query)
print(f"路由到: {route}")  # 可能输出: js_docs 或 python_docs
```

| 场景 | 无路由准确率 | 有路由准确率 | 检索时间 |
|-----|------------|------------|---------|
| 3个数据源 | 72% | 89% | -40% |
| 5个数据源 | 65% | 86% | -60% |
| 10个数据源 | 58% | 83% | -75% |

**结论：** 数据源越多，路由的价值越大。

### 5. 查询构建（Query Construction）

#### 原理
将自然语言查询转换为结构化查询语言（SQL、Cypher、元数据过滤器等），用于查询结构化数据或图数据库。

**通俗比喻：**
你去餐厅点菜，说"我要一份不辣的、素的、便宜的菜"。服务员把这翻译成：
```sql
SELECT * FROM menu 
WHERE spicy = false 
  AND vegetarian = true 
  AND price < 30
ORDER BY price ASC
```


#### 三种主要类型

**1. Text-to-SQL**
```
自然语言: "有多少个演员的年龄超过30岁？"
    ↓
SQL: SELECT COUNT(*) FROM actors WHERE age > 30;
```

**2. Text-to-Cypher（图数据库）**
```
自然语言: "孙悟空的师傅是谁？"
    ↓
Cypher: MATCH (悟空:角色 {name: "孙悟空"})-[:师徒]->(师傅) RETURN 师傅.name
```

**3. 元数据过滤器**
```
自然语言: "2023年发布的关于Python的视频"
    ↓
Filter: {
    "publish_year": 2023,
    "topic": "Python"
}
```

#### 工作流程（Text-to-SQL 示例）

```
用户查询: "销售额最高的前5个产品是什么？"
    ↓
LLM 分析查询意图
    ↓
理解数据库结构（表、字段、关系）
    ↓
生成 SQL:
SELECT product_name, SUM(sales) as total_sales
FROM sales
GROUP BY product_name
ORDER BY total_sales DESC
LIMIT 5;
    ↓
执行 SQL 查询
    ↓
返回结果
```

#### 关键技术点

**1. 数据库结构理解**
- 需要提供表结构（Schema）
- 字段说明和示例数据
- 表之间的关系

**2. 查询验证**
- SQL 语法检查
- 安全性检查（防止 SQL 注入）
- 权限检查

**3. 错误处理**
- SQL 执行失败时的重试机制
- 错误信息反馈给 LLM 重新生成


#### 优点
- ✅ 精确查询（比向量检索更准确）
- ✅ 支持复杂查询（聚合、统计、关联）
- ✅ 适合结构化数据
- ✅ 可以处理数值比较、范围查询

#### 缺点
- ❌ 需要结构化数据库
- ❌ LLM 可能生成错误的 SQL
- ❌ 需要理解数据库结构
- ❌ 不适合非结构化文本

#### 使用场景
- 数据库问答系统
- 商业智能报表
- 数据分析助手
- 企业数据查询
- 知识图谱问答

#### 代码示例

### 6. 查询澄清（Query Clarification）

#### 原理
当查询存在歧义或不明确时，通过与用户交互来澄清真实意图。

**通俗比喻：**
你去餐厅说"我要一份面"，服务员会问："您要牛肉面、鸡蛋面还是素面？"

#### 工作流程
```
用户查询: "悟空"
    ↓
系统识别歧义
    ↓
生成澄清问题:
  1. 您是想了解孙悟空的技能？
  2. 您是想了解孙悟空的背景故事？
  3. 您是想了解孙悟空的装备？
    ↓
用户选择: 1
    ↓
精确查询: "孙悟空的技能介绍"
    ↓
检索并返回结果
```

#### 适用场景
- 简短模糊的查询
- 多义词查询
- 交互式问答系统
- 对话式搜索

#### 优缺点
- ✅ 提高查询准确性
- ✅ 改善用户体验
- ❌ 增加交互轮次
- ❌ 需要用户参与

### 7. 查询扩展 - PRF

#### 原理
PRF (Pseudo-Relevance Feedback) 伪相关反馈：先用原始查询检索，然后从检索结果中提取关键词，扩展原始查询，再次检索。

**通俗比喻：**
你搜索"机器学习"，系统返回一些结果。你发现结果中经常出现"神经网络"、"深度学习"等词，于是系统自动把这些词加入你的查询，再搜索一次。


#### 工作流程
```
原始查询: "机器学习"
    ↓
第一次检索（Top 5 结果）
    ↓
提取高频关键词: "神经网络", "深度学习", "算法"
    ↓
扩展查询: "机器学习 神经网络 深度学习 算法"
    ↓
第二次检索
    ↓
返回更全面的结果
```

#### 与 HyDE 的区别

| 特性 | HyDE | PRF |
|-----|------|-----|
| **扩展方式** | LLM 生成假设文档 | 从检索结果提取关键词 |
| **检索次数** | 1次 | 2次 |
| **成本** | 高（LLM调用） | 低（关键词提取） |
| **效果** | 更好（语义丰富） | 较好（关键词匹配） |

#### 适用场景
- 成本敏感的场景
- 不需要 LLM 的场景
- 传统搜索引擎优化

### 8. 查询压缩（Query Compression）

#### 原理
在多轮对话中，将历史对话上下文和当前查询压缩成一个独立的查询，使其包含所有必要信息。

**通俗比喻：**
对话：
- 用户："孙悟空有哪些技能？"
- 系统："金箍棒、筋斗云..."
- 用户："第一个怎么用？"（指代不明确）

压缩后的查询："金箍棒技能怎么使用？"

#### 工作流程
```
对话历史:
  用户: "孙悟空有哪些技能？"
  助手: "金箍棒、筋斗云、七十二变"
  用户: "第一个怎么升级？"
    ↓
LLM 压缩
    ↓
独立查询: "金箍棒技能如何升级？"
    ↓
检索并返回结果
```

#### 适用场景
- 多轮对话系统
- 聊天机器人
- 对话式搜索


#### 代码示例
```python
from langchain.chains import create_history_aware_retriever
from langchain_core.prompts import MessagesPlaceholder

# 创建历史感知检索器
contextualize_q_prompt = ChatPromptTemplate.from_messages([
    ("system", "根据对话历史和最新问题，生成一个独立的查询。"),
    MessagesPlaceholder("chat_history"),
    ("human", "{input}"),
])

history_aware_retriever = create_history_aware_retriever(
    llm, retriever, contextualize_q_prompt
)

# 使用
chat_history = [
    HumanMessage(content="孙悟空有哪些技能？"),
    AIMessage(content="金箍棒、筋斗云、七十二变"),
]

result = history_aware_retriever.invoke({
    "chat_history": chat_history,
    "input": "第一个怎么升级？"
})
# 自动压缩为: "金箍棒技能如何升级？"
```

## 如何选择检索前处理技术？

### 决策树

```
开始
 ↓
查询是否口语化/有噪音？
 ├─ 是 → 使用查询重写
 └─ 否 ↓
    查询是否包含多个问题？
     ├─ 是 → 使用查询分解
     └─ 否 ↓
        查询是否过短（<5字）？
         ├─ 是 → 使用 HyDE 扩展
         └─ 否 ↓
            是否有多个数据源？
             ├─ 是 → 使用查询路由
             └─ 否 ↓
                是否查询结构化数据？
                 ├─ 是 → 使用查询构建（SQL/Cypher）
                 └─ 否 → 直接检索
```


### 场景推荐

| 场景 | 推荐技术 | 理由 |
|-----|---------|------|
| 客服系统 | 查询重写 + 路由 | 处理口语化，分流到不同部门 |
| 搜索引擎 | 查询重写 + HyDE | 标准化查询，提高准确率 |
| 复杂问答 | 查询分解 + 重写 | 处理复合问题 |
| 数据分析 | 查询构建（SQL） | 精确查询结构化数据 |
| 知识图谱 | 查询构建（Cypher） | 查询关系数据 |
| 多领域系统 | 查询路由 | 提高效率和准确率 |
| 对话系统 | 查询压缩 | 处理上下文依赖 |
| 简短查询 | HyDE | 扩展语义信息 |

### 组合使用建议

**最佳实践流程：**
```
1. 查询重写（去噪、标准化）
   ↓
2. 查询路由（选择数据源）
   ↓
3. 查询分解（如果是复合查询）
   ↓
4. HyDE 扩展（如果查询过短）
   ↓
5. 执行检索
```

**示例：**
```python
# 完整的检索前处理流程
def preprocess_and_retrieve(query: str):
    # 1. 查询重写
    rewritten_query = rewrite_query(query)
    
    # 2. 查询路由
    datasource = route_query(rewritten_query)
    
    # 3. 判断是否需要分解
    if is_complex_query(rewritten_query):
        sub_queries = decompose_query(rewritten_query)
        results = []
        for sub_q in sub_queries:
            # 4. 判断是否需要 HyDE
            if len(sub_q) < 10:
                sub_q = hyde_expand(sub_q)
            results.extend(retrieve(sub_q, datasource))
        return merge_results(results)
    else:
        # 4. 判断是否需要 HyDE
        if len(rewritten_query) < 10:
            rewritten_query = hyde_expand(rewritten_query)
        return retrieve(rewritten_query, datasource)
```

### 查询分解

**子查询数量控制：**
```python
# 在提示词中限制
"请将查询分解为 2-4 个子查询"

# 或在代码中过滤
sub_queries = decompose(query)
if len(sub_queries) > 5:
    sub_queries = sub_queries[:5]  # 只保留前5个
```

**去重策略：**
```python
# 使用语义去重而不是字符串去重
def semantic_dedup(docs, threshold=0.9):
    unique_docs = []
    for doc in docs:
        is_duplicate = False
        for unique_doc in unique_docs:
            if similarity(doc, unique_doc) > threshold:
                is_duplicate = True
                break
        if not is_duplicate:
            unique_docs.append(doc)
    return unique_docs
```

### HyDE

**假设文档长度：**
```python
# 短假设文档（100-200字）- 快速但信息少

# 中等假设文档（200-500字）- 平衡
"请详细回答，约300字..."

# 长假设文档（500+字）- 慢但信息丰富
"请非常详细地回答..."
```

prompts = [

### 查询路由

**置信度阈值：**
```python
# 高置信度 - 单路由
if confidence > 0.8:
    return [best_route]

elif confidence > 0.6:
    return [best_route, second_best_route]

# 缓存常见查询的路由结果
route_cache = {}

def route_with_cache(query: str):
    # 使用查询的哈希或向量作为缓存键
    cache_key = hash(query)
    
    if cache_key in route_cache:
        return route_cache[cache_key]
    
    route = route_query(query)
    route_cache[cache_key] = route
    return route
```

### Q5: 多个技术组合时的顺序重要吗？

**答：** 非常重要！推荐顺序：

```
1. 查询重写（清理噪音）
   ↓
2. 查询路由（选择数据源）
   ↓
3. 查询分解（处理复合查询）
   ↓
4. HyDE 扩展（增强语义）
   ↓
5. 查询构建（转换为结构化查询，如果需要）
```

**错误顺序示例：**
```
❌ 先分解再重写 → 可能分解出有噪音的子查询
❌ 先路由再重写 → 可能因为噪音导致路由错误
```

def multilingual_preprocessing(query: str):

    lang = detect_language(query)
    
    # 如果不是英语，翻译

def customer_service_pipeline(query: str):

## 总结与建议

### 核心要点

1. **没有银弹**：没有一种技术适用于所有场景，需要根据实际情况选择
2. **组合使用**：多种技术组合通常效果最好，但要注意成本和延迟
3. **持续优化**：通过 A/B 测试和用户反馈不断优化
4. **成本意识**：每次 LLM 调用都有成本，要权衡效果和成本

### 快速决策指南

| 如果你的查询... | 使用... | 预期提升 |
|---------------|--------|---------|
| 口语化、有噪音 | 查询重写 | +30-40% |
| 包含多个问题 | 查询分解 | +40-60% |
| 过短（<5字） | HyDE | +25-35% |
| 涉及多个领域 | 查询路由 | +40-50% |
| 查询结构化数据 | 查询构建 | +60-80% |
| 存在歧义 | 查询澄清 | +20-30% |
| 在对话中 | 查询压缩 | +15-25% |
