## 前言

前面我们学习了向量存储和 Milvus 的基本使用，知道如何存储和检索向量。但有个关键问题：当数据量达到百万、千万级别时，如何保证搜索速度？

答案就是向量索引。就像图书馆需要索引系统才能快速找书，向量数据库也需要索引才能快速找到相似向量。

这篇文章我们深入学习各种索引算法的原理、特点和使用场景。

---

## 什么是向量索引？

想象你有一个巨大的图书馆，里面有上百万本书。如果没有索引系统（比如分类、目录），你要找一本书就得一本一本翻，效率极低。

向量索引就是为向量数据库建立的"图书馆索引系统"。它通过特殊的数据结构，让你能快速找到与查询向量最相似的向量，而不需要逐个比较所有向量。

**核心权衡：**
- **准确率 vs 速度**：越精确的搜索越慢，越快的搜索可能牺牲一些准确率
- **内存 vs 磁盘**：内存索引快但贵，磁盘索引慢但能处理海量数据
- **构建时间 vs 查询时间**：有些索引构建慢但查询快，有些相反

---

## 索引算法对比总览

| 索引类型 | 搜索速度 | 准确率 | 内存占用 | 适用数据量 | 构建速度 | 推荐场景 |
|---------|---------|--------|---------|-----------|---------|---------|
| FLAT | 慢 (1/5) | 100% 完美 | 高 | < 10万 | 快 | 小数据集、需要完美准确率 |
| IVF_FLAT | 中等 (3/5) | 高 (95-99%) | 中等 | 10万-100万 | 中等 | 平衡性能和准确率 |
| IVF_PQ | 快 (4/5) | 中等 (90-95%) | 低 | 100万-1000万 | 中等 | 大数据集、内存受限 |
| HNSW | 非常快 (5/5) | 高 (95-99%) | 高 | 10万-1000万 | 慢 | 高性能查询、内存充足 |
| DiskANN | 快 (4/5) | 高 (95-99%) | 极低 | 1000万+ | 慢 | 超大数据集、磁盘存储 |

## 详细算法解析

### 1. FLAT 索引（暴力搜索）

#### 原理
FLAT 是最简单的索引方式，实际上就是"不建索引"。它会把查询向量和数据库中的每一个向量都计算一遍距离，然后返回最近的 K 个。

**通俗比喻：**
就像在一个没有分类的书架上找书，你必须从第一本看到最后一本，逐个检查书名。

#### 工作流程
```
查询向量: [0.1, 0.5, 0.3, ...]
         ↓
计算距离: 向量1 → 距离 0.8
         向量2 → 距离 0.3  ← 最近
         向量3 → 距离 0.9
         ...
         向量N → 距离 0.5
         ↓
返回最近的 K 个
```

#### 优点
- 100% 准确率，保证找到真正最近的向量
- 不需要构建索引，插入数据快
- 实现简单，没有参数需要调优

#### 缺点
- 搜索速度慢，时间复杂度 O(N)
- 数据量大时完全不可用
- 内存占用高（需要加载所有向量）

#### 使用场景
- 数据量小于 10 万条
- 需要 100% 准确率的场景（如科研、基准测试）
- 原型开发和测试阶段

#### 代码示例
```python
index_params.add_index(
    field_name="vector",
    metric_type="L2",
    index_type="FLAT",
    params={}  # 无需参数
)
```

### 2. IVF_FLAT 索引（倒排文件索引）

#### 原理
IVF (Inverted File) 通过聚类算法将向量空间分成多个区域（称为"桶"或"聚类中心"），搜索时只在最相关的几个区域内查找。

**通俗比喻：**
把图书馆的书分成"科技"、"文学"、"历史"等类别。找书时，先确定是哪个类别，然后只在那个类别的书架上找，而不是整个图书馆都翻一遍。

#### 工作流程
```
构建阶段:
所有向量 → K-means 聚类 → 分成 nlist 个聚类
         [聚类1: 向量1, 向量5, ...]
         [聚类2: 向量2, 向量7, ...]
         [聚类3: 向量3, 向量9, ...]

搜索阶段:
查询向量 → 找到最近的 nprobe 个聚类中心
         → 只在这些聚类内暴力搜索
         → 返回最近的 K 个向量
```

#### 关键参数
- **nlist**（构建参数）：聚类中心数量
  - 推荐值：`4 * sqrt(N)`，N 是向量总数
  - 例如：100 万向量 → nlist ≈ 4000
  
- **nprobe**（搜索参数）：搜索时检查的聚类数量
  - 范围：1 到 nlist
  - 越大越准确但越慢
  - 推荐：nlist 的 5-10%

#### 优点
- 比 FLAT 快 10-100 倍
- 准确率高（95-99%）
- 参数简单易调

#### 缺点
- 需要构建索引时间
- 内存占用仍然较高（存储完整向量）
- 数据分布不均时效果差

#### 使用场景
- 10 万到 100 万条数据
- 需要平衡速度和准确率
- 内存充足的场景

#### 代码示例
```python
index_params.add_index(
    field_name="vector",
    index_type="IVF_FLAT",
    params={
        "nlist": 1024  # 聚类数量
    }
)

# 搜索时
search_params = {
    "params": {"nprobe": 64}  # 检查 64 个聚类
}
```

### 3. IVF_PQ 索引（乘积量化）

#### 原理
在 IVF_FLAT 的基础上，使用乘积量化（Product Quantization）压缩向量。把每个向量切成多个子向量，每个子向量用一个编码代替，大幅减少内存占用。

**通俗比喻：**
不仅把书分类（IVF），还把每本书的内容压缩成摘要（PQ）。虽然摘要不如原文详细，但占用空间小得多，查找时先用摘要筛选，需要时再看原文。

#### 工作流程
```
原始向量: [0.1, 0.5, 0.3, 0.8, 0.2, 0.7, 0.4, 0.9]  (128维)
         ↓ 切分成 m 个子向量
子向量1: [0.1, 0.5, 0.3, 0.8]  → 编码: 25
子向量2: [0.2, 0.7, 0.4, 0.9]  → 编码: 142
         ↓
压缩后: [25, 142]  (只需 2 个字节，而不是 128*4=512 字节)
```

#### 关键参数
- **nlist**：聚类数量（同 IVF_FLAT）
- **m**：子向量数量
  - 必须能被向量维度整除
  - 推荐：dim / m ≥ 2
  - 例如：128 维 → m = 32 或 64
  
- **nbits**：每个子向量的编码位数
  - 通常固定为 8（即 256 个码本）

#### 优点
- 内存占用极低（压缩 8-32 倍）
- 搜索速度快
- 可处理更大数据集

#### 缺点
- 准确率下降（90-95%）
- 构建索引时间长
- 参数调优复杂

#### 使用场景
- 100 万到 1000 万条数据
- 内存受限的环境
- 可以接受略微降低的准确率

#### 代码示例
```python
index_params.add_index(
    field_name="vector",
    index_type="IVF_PQ",
    params={
        "nlist": 1024,  # 聚类数量
        "m": 32,        # 子向量数量（128/32=4）
        "nbits": 8      # 编码位数
    }
)
```

### 4. HNSW 索引（分层导航小世界图）

#### 原理
HNSW (Hierarchical Navigable Small World) 构建一个多层图结构，每个向量是图中的节点，相似的向量之间有边连接。搜索时从顶层快速定位到大致区域，然后逐层下降精确查找。

**通俗比喻：**
像高速公路系统：
- 顶层：国道（连接城市，跨度大）
- 中层：省道（连接区县，跨度中等）
- 底层：市内道路（连接街道，精确定位）

找目的地时，先上高速快速接近，然后下高速走省道，最后走市内道路精确到达。

#### 工作流程
```
构建阶段:
层级 2: A ←→ B ←→ C  (稀疏，长距离连接)
         ↓     ↓     ↓
层级 1: A ←→ D ←→ B ←→ E ←→ C  (中等密度)
         ↓  ↓  ↓  ↓  ↓  ↓  ↓
层级 0: A-D-F-B-G-E-H-C-I  (密集，短距离连接)

搜索阶段:
查询向量 → 从顶层入口开始
         → 贪心搜索找到该层最近邻
         → 下降到下一层
         → 重复直到底层
         → 返回最近的 K 个
```

#### 关键参数
- **M**（构建参数）：每个节点的最大邻居数
  - 范围：4-64
  - 推荐：16-64
  - 越大越准确但内存占用越高
  
- **efConstruction**（构建参数）：构建时的候选邻居数
  - 范围：100-500
  - 越大构建越慢但质量越好
  
- **ef**（搜索参数）：搜索时的候选邻居数
  - 范围：top_k 到 512
  - 越大越准确但越慢

#### 优点
- 搜索速度极快（最快的内存索引）
- 准确率高（95-99%）
- 支持动态插入和删除
- 对数据分布不敏感

#### 缺点
- 内存占用高（比原始向量多 1.5-2 倍）
- 构建索引慢
- 不适合超大规模数据（内存限制）

#### 使用场景
- 10 万到 1000 万条数据
- 需要极致查询性能
- 内存充足的生产环境
- 需要频繁更新数据

#### 代码示例
```python
index_params.add_index(
    field_name="vector",
    index_type="HNSW",
    params={
        "M": 64,              # 最大邻居数
        "efConstruction": 200  # 构建时候选数
    }
)

# 搜索时
search_params = {
    "params": {"ef": 128}  # 搜索时候选数
}
```

### 5. DiskANN 索引（磁盘索引）

#### 原理
DiskANN 是微软开发的基于磁盘的索引算法，通过精心设计的图结构和缓存策略，实现在磁盘上也能高效搜索。它将热点数据缓存在内存，冷数据存在磁盘。

**通俗比喻：**
像一个智能仓库系统：
- 常用的商品放在前台（内存缓存）
- 不常用的商品放在后仓（磁盘）
- 通过智能预测，提前把可能需要的商品调到前台

#### 工作流程
```
构建阶段:
向量数据 → 构建 Vamana 图（优化的导航图）
         → 热点节点缓存到内存
         → 冷数据存储到 SSD

搜索阶段:
查询向量 → 从内存缓存的入口开始
         → 图导航（部分访问磁盘）
         → 智能预取减少磁盘 I/O
         → 返回最近的 K 个
```

#### 关键参数
- **search_list**（搜索参数）：搜索时的候选列表大小
  - 范围：top_k 到 200
  - 越大越准确但磁盘 I/O 越多

#### 优点
- 内存占用极低（只需 GB 级内存处理 TB 级数据）
- 可处理超大规模数据（亿级以上）
- 准确率高（95-99%）
- 成本低（用便宜的磁盘代替昂贵的内存）

#### 缺点
- 需要高速 SSD（机械硬盘不适用）
- 构建索引时间长
- 搜索速度比纯内存索引慢
- 不适合频繁更新

#### 使用场景
- 1000 万条以上数据
- 内存预算有限
- 数据更新不频繁
- 有高速 SSD 存储

#### 代码示例
```python
index_params.add_index(
    field_name="vector",
    index_type="DISKANN",
    params={}  # 使用默认参数
)

# 搜索时
search_params = {
    "params": {"search_list": 64}
}
```

## 其他常用索引算法

### 6. SCANN 索引

**来源：** Google 开发

**原理：** 结合量化和哈希技术，使用学习到的量化器优化搜索性能。

**特点：**
- 在准确率和速度之间取得很好的平衡
- 特别适合高维向量（>100 维）
- Google 内部大规模使用

**Milvus 支持情况：** 目前不直接支持，但可以通过 FAISS 后端使用类似技术

---

### 7. ANNOY 索引

**来源：** Spotify 开发

**原理：** 构建多棵随机投影树（Random Projection Trees），搜索时查询多棵树并合并结果。

**通俗比喻：**
建立多个不同的分类系统（比如按作者、按年代、按主题），查找时同时在多个系统中搜索，然后合并结果。

**特点：**
- 构建快，内存占用低
- 支持 mmap，可以多进程共享索引
- 适合静态数据（不支持动态更新）

**Milvus 支持情况：** 不直接支持

**适用场景：**
- 推荐系统
- 静态数据集
- 需要多进程共享索引

---

### 8. FAISS GPU 索引

**来源：** Facebook AI Research

**原理：** 利用 GPU 的并行计算能力加速向量搜索，支持多种索引类型（IVF、PQ、HNSW 等）的 GPU 版本。

**特点：**
- 搜索速度极快（比 CPU 快 10-100 倍）
- 支持批量搜索优化
- 需要 NVIDIA GPU 和 CUDA

**Milvus 支持情况：** 支持，通过配置可以使用 GPU 加速

**适用场景：**
- 有 GPU 资源
- 需要极致性能
- 批量查询场景

## 如何选择索引算法？

### 决策树

```
开始
 ↓
数据量 < 10万？
 ├─ 是 → 使用 FLAT（追求完美准确率）
 └─ 否 ↓
    内存充足？
     ├─ 是 → 数据量 < 1000万？
     │        ├─ 是 → 使用 HNSW（最佳性能）
     │        └─ 否 → 使用 DiskANN（超大规模）
     └─ 否 ↓
        数据量 < 100万？
         ├─ 是 → 使用 IVF_FLAT（平衡选择）
         └─ 否 → 使用 IVF_PQ（压缩存储）
```

### 场景推荐

| 场景 | 推荐索引 | 理由 |
|-----|---------|------|
| 原型开发/测试 | FLAT | 简单、准确、无需调参 |
| 小型生产系统（<100万） | IVF_FLAT | 性能和准确率平衡 |
| 中型系统（100万-1000万） | HNSW | 最佳查询性能 |
| 大型系统（>1000万） | DiskANN | 成本效益最优 |
| 内存受限 | IVF_PQ | 压缩比高 |
| 实时推荐系统 | HNSW | 低延迟 |
| 离线批处理 | IVF_FLAT | 吞吐量高 |
| 嵌入式设备 | IVF_PQ | 资源占用少 |

## 参数调优建议

### FLAT
无需调优，开箱即用。

### IVF_FLAT / IVF_PQ

**构建参数：**
```python
nlist = 4 * sqrt(N)  # N 是向量总数

# 示例
N = 1,000,000  → nlist = 4000
N = 10,000,000 → nlist = 12649
```

**搜索参数：**
```python
# 快速搜索（准确率 ~90%）
nprobe = nlist * 0.01  

# 平衡搜索（准确率 ~95%）
nprobe = nlist * 0.05  

# 高精度搜索（准确率 ~99%）
nprobe = nlist * 0.10  
```

### HNSW

**构建参数：**
```python
# 标准配置
M = 16
efConstruction = 200

# 高性能配置（更快搜索，更多内存）
M = 64
efConstruction = 500

# 节省内存配置
M = 8
efConstruction = 100
```

**搜索参数：**
```python
# 快速搜索
ef = top_k * 2

# 平衡搜索
ef = top_k * 4

# 高精度搜索
ef = top_k * 8
```

### DiskANN

**搜索参数：**
```python
# 快速搜索
search_list = 32

# 平衡搜索
search_list = 64

# 高精度搜索
search_list = 128
```
