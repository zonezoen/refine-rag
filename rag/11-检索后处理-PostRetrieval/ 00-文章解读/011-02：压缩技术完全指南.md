## 前言

前面我们学习了重排技术，知道如何提升检索结果的精度。但在实际应用中，检索到的文档往往很长，直接输入给LLM会带来高昂的成本和缓慢的响应速度。同时，冗长的文档还可能包含大量无关信息，影响LLM的理解和回答质量。

压缩技术就是解决这个问题的关键：去掉冗余信息，保留精华内容，让RAG系统既快又省钱，还能提供更准确的答案。

## 什么是压缩（Compression）？

### 用生活例子理解压缩

想象你要给朋友讲一个故事：

没有压缩（原始版本）：
```
"昨天我去了超市，超市里人很多，我在超市里转了很久，
最后在超市的水果区买了苹果，然后在超市的收银台结账，
最后从超市出来回家了。"
（60个字，但很啰嗦）
```

压缩后（精简版本）：
```
"昨天去超市买了苹果。"
（10个字，保留核心信息）
```

在RAG系统中的压缩就是这样！去掉冗余，保留精华。

## 为什么需要压缩？

### 三大核心原因

#### 1. 降低成本

```
场景：使用GPT-4处理检索结果

没有压缩：
- 检索到10个文档，每个500字 = 5000字
- GPT-4输入成本：$0.03/1K tokens
- 单次查询成本：约$0.15

使用压缩：
- 压缩到50% = 2500字
- 单次查询成本：约$0.075
- 节省50%成本！

每天1000个查询：
- 没压缩：$150/天 = $4500/月
- 有压缩：$75/天 = $2250/月
- 每月节省$2250！
```

#### 2. 提高速度

```
输入越少 → LLM处理越快

示例（GPT-4）：
- 5000 tokens输入：生成时间 ~8秒
- 2500 tokens输入：生成时间 ~4秒
- 速度提升50%！
```

#### 3. 提升质量

```
问题：信息过载

没有压缩：
用户问："五台山的开放时间？"
输入给LLM：
- 文档1：五台山的历史（500字）
- 文档2：五台山的地理位置（500字）
- 文档3：五台山的开放时间（50字）
- 文档4：五台山的门票价格（500字）
→ LLM被无关信息干扰，可能答非所问

使用压缩：
只保留相关部分：
- 文档3：五台山的开放时间（50字）
→ LLM聚焦核心信息，答案更准确
```

## 技术详解

### 1. ContextualCompression（上下文压缩）

**文件：** `01-ContextualCompressionRetriever压缩.py`

#### 核心思想

结合重排和压缩，只保留最相关的文档。

#### 通俗例子

你在图书馆找资料：

**传统方法：**
```
1. 找到10本相关的书
2. 把10本书全部带回家
3. 慢慢看哪些有用
→ 太重了！
```

**ContextualCompression方法：**
```
1. 找到10本相关的书
2. 快速翻阅，只复印最相关的3页
3. 只带这3页回家
→ 轻松多了！
```

#### 工作流程

```
用户查询："山西有哪些景点？"
    ↓
初步检索 → 10个文档
    ↓
Cohere重排 → 按相关性排序
    ↓
压缩 → 只保留top 3
    ↓
返回精简结果
```

#### 代码示例

```python
from langchain_cohere import CohereRerank
from langchain.retrievers.contextual_compression import ContextualCompressionRetriever

# 1. 创建基础检索器
retriever = BM25Retriever.from_documents(documents)

# 2. 创建Cohere重排器（同时做压缩）
compressor = CohereRerank(model="rerank-multilingual-v3.0")

# 3. 组合成压缩检索器
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=retriever
)

# 4. 一步到位：检索+重排+压缩
results = compression_retriever.invoke(query)
```

#### 优势
- 简单易用（一行代码搞定）
- 效果优秀（Cohere商业级模型）
- 自动重排+压缩（一石二鸟）
- 多语言支持（包括中文）

#### 劣势
- 需要API密钥（Cohere账号）
- 按量付费（有成本）
- 依赖网络（需要联网）

#### 适用场景
- 商业应用（预算充足）
- 快速原型开发
- 对效果要求高
- 不想自己维护模型

### 2. LLMLingua（智能提示词压缩）

**文件：** `02-LLMLingua压缩.py`

#### 核心思想

使用小型语言模型智能删除不重要的词，保留关键信息。

#### 通俗例子

**原文（啰嗦版）：**
```
"云冈石窟位于中国北部山西省大同市西郊17公里处的武周山南麓，
石窟依山开凿，东西绵延1公里。存有主要洞窟45个，大小窟龛252个，
石雕造像51000余躯，为中国规模最大的古代石窟群之一..."
（150字）
```

**LLMLingua压缩后：**
```
"云冈石窟 山西 大同 武周山 45洞窟 252窟龛 51000造像 
中国最大石窟群"
（30字，压缩80%）
```

就像发电报：只保留关键词，删除"的、了、是"等虚词。

#### 工作原理

```
步骤1：分词
"云冈石窟位于山西省" → ["云冈石窟", "位于", "山西省"]

步骤2：计算每个词的重要性
- "云冈石窟"：重要性 0.95 ✅
- "位于"：重要性 0.10 ❌
- "山西省"：重要性 0.90 ✅

步骤3：删除低重要性的词
保留："云冈石窟 山西省"
```

#### 代码示例

```python
from llmlingua import PromptCompressor

# 1. 初始化压缩器
compressor = PromptCompressor(
    model_name="NousResearch/Llama-2-7b-hf",
    device_map="cpu"  # 使用CPU
)

# 2. 压缩文本
compressed = compressor.compress_prompt(
    context="云冈石窟位于中国北部山西省...",
    instruction="压缩并保持主要内容",
    target_token=50  # 目标：压缩到50个token
)

# 3. 获取压缩结果
print(compressed['compressed_prompt'])
print(f"压缩率: {compressed['rate']}")
```

#### 压缩率控制

```python
# 方式1：指定目标token数
compressed = compressor.compress_prompt(
    context=text,
    target_token=100  # 压缩到100个token
)

# 方式2：指定压缩比例
compressed = compressor.compress_prompt(
    context=text,
    rate=0.5  # 保留50%
)
```

#### JSON数据压缩

LLMLingua还支持结构化数据压缩：

```python
json_data = {
    "id": 12345,
    "name": "张三",
    "biography": "张三是一位优秀的工程师，毕业于清华大学..."
}

# 配置每个字段的压缩策略
json_config = {
    "id": {
        "rate": 1.0,           # 不压缩
        "compress": False
    },
    "name": {
        "rate": 0.8,           # 轻度压缩
        "compress": False
    },
    "biography": {
        "rate": 0.3,           # 重度压缩
        "compress": True
    }
}

compressed_json = compressor.compress_json(json_data, json_config)
```

#### 优势
- ✅ 压缩率极高（可达90%）
- ✅ 开源免费
- ✅ 支持多种语言
- ✅ 可本地部署

#### 劣势
- ❌ 需要下载模型（7GB）
- ❌ 压缩质量依赖模型
- ❌ 可能丢失重要信息
- ❌ 压缩后可读性差

#### 适用场景
- 需要极致压缩（成本敏感）
- 输入token限制严格
- 有本地计算资源
- 对可读性要求不高

#### 压缩效果对比

```
原文（200 tokens）：
"云冈石窟位于中国北部山西省大同市西郊17公里处的武周山南麓，
石窟依山开凿，东西绵延1公里。存有主要洞窟45个，大小窟龛252个，
石雕造像51000余躯，为中国规模最大的古代石窟群之一，与敦煌莫高窟、
洛阳龙门石窟和天水麦积山石窟并称为中国四大石窟艺术宝库。"

压缩50%（100 tokens）：
"云冈石窟 中国 山西省 大同市 武周山 石窟 开凿 1公里 
45洞窟 252窟龛 51000造像 中国最大 石窟群 敦煌莫高窟 
洛阳龙门石窟 麦积山石窟 四大石窟"

压缩80%（40 tokens）：
"云冈石窟 山西 大同 45洞窟 51000造像 四大石窟"
```

#### 注意事项

1. **压缩率不是越高越好**
```python
# ❌ 过度压缩
rate=0.1  # 只保留10%，信息丢失严重

# ✅ 适度压缩
rate=0.5  # 保留50%，平衡信息和成本
```

2. **重要字段不要压缩**
```python
# 对于关键信息（ID、名称等）
"id": {"rate": 1.0, "compress": False}
```

3. **测试压缩效果**
```python
# 压缩前后对比测试
original_answer = llm(original_context + query)
compressed_answer = llm(compressed_context + query)
# 比较答案质量
```

### 3️⃣ SentenceEmbeddingOptimizer（句子级优化）

**文件：** `03-SentenceEmbeddingOptimizer压缩.py`

#### 核心思想

基于句子的语义相似度，只保留与查询最相关的句子。

#### 通俗例子

你问："五台山的开放时间？"

**文档内容：**
```
句子1："五台山位于山西省忻州市。" (相似度: 0.3)
句子2："五台山是佛教名山。" (相似度: 0.4)
句子3："五台山全年开放，6:00-18:00。" (相似度: 0.9) ✅
句子4："五台山门票价格135元。" (相似度: 0.5)
```

**SentenceEmbedding只保留：**
```
句子3："五台山全年开放，6:00-18:00。" ✅
```

就像用筛子筛选：只留下最相关的句子。

#### 工作原理

```
步骤1：文档分句
文档 → ["句子1", "句子2", "句子3", ...]

步骤2：计算相似度
for 每个句子:
    相似度 = cosine_similarity(查询向量, 句子向量)

步骤3：过滤
方式A：百分位截断
  保留相似度前50%的句子

方式B：阈值截断
  保留相似度 > 0.7的句子
```

#### 代码示例

```python
from llama_index.core.postprocessor import SentenceEmbeddingOptimizer

# 方式1：百分位截断（保留前50%）
optimizer = SentenceEmbeddingOptimizer(
    percentile_cutoff=0.5  # 保留相似度前50%的句子
)

query_engine = index.as_query_engine(
    node_postprocessors=[optimizer]
)

# 方式2：阈值截断（保留相似度>0.7的句子）
optimizer = SentenceEmbeddingOptimizer(
    threshold_cutoff=0.7  # 只保留相似度>0.7的句子
)
```

#### 两种截断方式对比

**百分位截断（percentile_cutoff）：**
```python
percentile_cutoff=0.5  # 保留前50%

优点：
- 总是返回固定比例的内容
- 适合需要稳定输出长度的场景

缺点：
- 可能保留不相关的句子（如果整体相关性都低）
- 可能丢弃相关的句子（如果整体相关性都高）

示例：
10个句子，相似度：[0.9, 0.85, 0.8, 0.3, 0.25, 0.2, 0.15, 0.1, 0.05, 0.02]
保留前5个：[0.9, 0.85, 0.8, 0.3, 0.25]
→ 0.3和0.25其实不太相关，但也被保留了
```

**阈值截断（threshold_cutoff）：**
```python
threshold_cutoff=0.7  # 只保留>0.7的

优点：
- 保证质量（只保留真正相关的）
- 自动适应不同查询

缺点：
- 输出长度不稳定
- 可能返回很少或很多内容

示例：
10个句子，相似度：[0.9, 0.85, 0.8, 0.3, 0.25, 0.2, 0.15, 0.1, 0.05, 0.02]
保留>0.7的：[0.9, 0.85, 0.8]
→ 只保留真正相关的3个句子
```

#### 优势
- 精确过滤（句子级别）
- 速度快（只需计算相似度）
- 开源免费
- 易于理解和调试

#### 劣势
- 可能破坏上下文连贯性
- 需要好的句子分割
- 依赖嵌入模型质量

#### 适用场景
- 文档结构清晰（句子独立）
- 需要精确匹配
- 对上下文连贯性要求不高
- 快速原型开发

## 如何选择压缩技术？

### 决策树

```
开始
  ↓
是否有预算？
  ├─ 有 → ContextualCompression（最省心）
  └─ 无 ↓
       ↓
需要极致压缩（>70%）？
  ├─ 是 → LLMLingua
  └─ 否 ↓
       ↓
文档结构清晰（句子独立）？
  ├─ 是 → SentenceEmbedding
  └─ 否 → LLMLingua（更通用）
```


```
推荐：LLMLingua
理由：
- 完全免费
- 压缩率高（节省成本）
- 可本地部署
```

#### 场景3：FAQ系统
```
推荐：SentenceEmbedding
理由：
- 问答对独立
- 精确匹配
- 速度快
```

#### 场景4：长文档摘要
```
推荐：LLMLingua
理由：
- 支持极长文本
- 保留关键信息
- 压缩率可控
```

## 实战：组合使用多种压缩技术

### 多阶段压缩Pipeline

```python
# 第一阶段：粗压缩（SentenceEmbedding）
# 快速过滤明显不相关的句子
optimizer = SentenceEmbeddingOptimizer(threshold_cutoff=0.5)
docs = optimizer.filter(docs)
# 1000句 → 500句

# 第二阶段：精压缩（ContextualCompression）
# 精确重排和压缩
compressor = CohereRerank()
docs = compressor.compress(docs)
# 500句 → 100句

# 第三阶段：极致压缩（LLMLingua，可选）
# 如果还需要进一步压缩
if len(docs) > token_limit:
    llm_lingua = PromptCompressor()
    docs = llm_lingua.compress(docs, rate=0.5)
    # 100句 → 50句
```

### 自适应压缩策略

```python
def adaptive_compress(docs, query, budget):
    """
    根据预算自适应选择压缩策略
    
    Args:
        docs: 文档列表
        query: 查询
        budget: 预算（美元）
    """
    total_tokens = count_tokens(docs)
    
    if budget > 0.1:
        # 预算充足：使用Cohere
        compressor = CohereRerank()
        return compressor.compress(docs, query)
    
    elif total_tokens > 10000:
        # token太多：使用LLMLingua极致压缩
        compressor = PromptCompressor()
        return compressor.compress(docs, rate=0.3)
    
    else:
        # 默认：使用SentenceEmbedding
        optimizer = SentenceEmbeddingOptimizer(threshold_cutoff=0.6)
        return optimizer.filter(docs, query)
```


## 重要知识点

### 1. 抽取式摘要 vs 生成式摘要

#### 抽取式摘要（Extractive Summarization）

**定义：** 从原文中选择重要的句子组成摘要。

**特点：**
- ✅ 保留原文表达
- ✅ 不会产生错误信息
- ❌ 可能不够流畅
- ❌ 受限于原文句子

**示例：**
```
原文：
"云冈石窟位于山西大同。石窟有45个洞窟。
石窟是世界文化遗产。石窟建于北魏时期。"

抽取式摘要：
"云冈石窟位于山西大同。石窟是世界文化遗产。"
（直接选择原文句子）
```

**实现方法：**
```python
# TextRank算法
from summa import summarizer

summary = summarizer.summarize(text, ratio=0.3)  # 保留30%

# LexRank算法
from sumy.parsers.plaintext import PlaintextParser
from sumy.nlp.tokenizers import Tokenizer
from sumy.summarizers.lex_rank import LexRankSummarizer

parser = PlaintextParser.from_string(text, Tokenizer("chinese"))
summarizer = LexRankSummarizer()
summary = summarizer(parser.document, 3)  # 3个句子
```

#### 生成式摘要（Abstractive Summarization）

**定义：** 理解原文后用新的语言重新表达。

**特点：**
- ✅ 更流畅自然
- ✅ 可以改写和概括
- ❌ 可能产生幻觉
- ❌ 计算成本高

**示例：**
```
原文：
"云冈石窟位于山西大同。石窟有45个洞窟。
石窟是世界文化遗产。石窟建于北魏时期。"

生成式摘要：
"山西大同的云冈石窟是北魏时期建造的世界文化遗产，
包含45个洞窟。"
（重新组织和表达）
```

**实现方法：**
```python
# 使用LLM生成摘要
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4")
prompt = f"请用一句话总结：{text}"
summary = llm.invoke(prompt)

# 使用专门的摘要模型
from transformers import pipeline

summarizer = pipeline("summarization", model="facebook/bart-large-cnn")
summary = summarizer(text, max_length=100)
```

#### 何时使用哪种？

```
抽取式摘要：
- 需要保证准确性（法律、医疗）
- 原文表达已经很好
- 计算资源有限

生成式摘要：
- 需要流畅的表达
- 原文冗长啰嗦
- 有充足的计算资源
```

### 2. 分块压缩（Chunk-level Compression）

#### 什么是分块压缩？

在文档切块后，对每个块单独压缩。

#### 为什么需要？

```
问题：整体压缩可能丢失重要信息

示例：
文档有10个段落，只有第3段和第7段相关
如果整体压缩50%，可能两个段落都被压缩掉了

解决：分块压缩
- 对每个段落单独评分
- 保留相关段落的完整内容
- 删除不相关段落
```

#### 实现方法

```python
def chunk_level_compress(chunks, query, threshold=0.7):
    """
    分块级别的压缩
    """
    compressed_chunks = []
    
    for chunk in chunks:
        # 计算每个块与查询的相关性
        relevance = calculate_relevance(chunk, query)
        
        if relevance > threshold:
            # 相关的块：保留完整内容
            compressed_chunks.append(chunk)
        elif relevance > threshold * 0.5:
            # 部分相关：压缩50%
            compressed = compress(chunk, rate=0.5)
            compressed_chunks.append(compressed)
        # 不相关的块：直接丢弃
    
    return compressed_chunks
```

### 3. 渐进式压缩（Progressive Compression）

#### 什么是渐进式压缩？

根据token预算动态调整压缩率。

#### 工作流程

```python
def progressive_compress(docs, max_tokens):
    """
    渐进式压缩：逐步增加压缩率直到满足token限制
    """
    current_tokens = count_tokens(docs)
    compression_rate = 1.0  # 初始不压缩
    
    while current_tokens > max_tokens:
        # 逐步增加压缩率
        compression_rate -= 0.1
        
        if compression_rate < 0.2:
            # 最多压缩到20%
            break
        
        # 应用压缩
        compressed_docs = compress(docs, rate=compression_rate)
        current_tokens = count_tokens(compressed_docs)
    
    return compressed_docs
```

#### 优势

- 自动适应token限制
- 尽可能保留信息
- 避免过度压缩

### 4. 选择性压缩（Selective Compression）

#### 什么是选择性压缩？

对不同类型的内容使用不同的压缩策略。

#### 策略示例

```python
def selective_compress(document):
    """
    选择性压缩：根据内容类型选择策略
    """
    compressed_parts = []
    
    for part in document.parts:
        if part.type == "title":
            # 标题：不压缩
            compressed_parts.append(part)
        
        elif part.type == "key_fact":
            # 关键事实：轻度压缩
            compressed = compress(part, rate=0.8)
            compressed_parts.append(compressed)
        
        elif part.type == "example":
            # 示例：中度压缩
            compressed = compress(part, rate=0.5)
            compressed_parts.append(compressed)
        
        elif part.type == "background":
            # 背景信息：重度压缩
            compressed = compress(part, rate=0.3)
            compressed_parts.append(compressed)
    
    return compressed_parts
```

### 6. 缓存压缩结果

#### 为什么需要缓存？

```
问题：重复压缩浪费资源

场景：
- 用户A问："五台山开放时间？"
- 用户B问："五台山几点开门？"
→ 两个问题检索到相同文档
→ 重复压缩浪费时间和成本

解决：缓存压缩结果
```

#### 实现方法

```python
import hashlib
from functools import lru_cache

class CompressionCache:
    def __init__(self):
        self.cache = {}
    
    def get_cache_key(self, text, compression_params):
        """生成缓存键"""
        content = text + str(compression_params)
        return hashlib.md5(content.encode()).hexdigest()
    
    def compress_with_cache(self, text, compressor, params):
        """带缓存的压缩"""
        cache_key = self.get_cache_key(text, params)
        
        if cache_key in self.cache:
            print("使用缓存结果")
            return self.cache[cache_key]
        
        # 执行压缩
        compressed = compressor.compress(text, **params)
        
        # 存入缓存
        self.cache[cache_key] = compressed
        
        return compressed

# 使用示例
cache = CompressionCache()
compressed = cache.compress_with_cache(
    text=document,
    compressor=llm_lingua,
    params={"rate": 0.5}
)
```

```python
    for table in doc.tables:
```

## 性能对比实验

### 实验设置

```
数据集：100个查询，每个查询检索10个文档
文档平均长度：500 tokens
评估指标：
- 压缩率
- 答案质量（F1 score）
- 处理时间
- 成本
```

### 实验结果

| 方法 | 压缩率 | 答案质量 | 处理时间 | 成本/100查询 |
|------|--------|----------|----------|--------------|
| 无压缩 | 0% | 0.85 | 5s | $15 |
| ContextualCompression | 60% | 0.83 | 8s | $8 |
| LLMLingua | 70% | 0.79 | 12s | $6 |
| SentenceEmbedding | 50% | 0.81 | 6s | $7.5 |

### 结论

1. **压缩率 vs 质量**：压缩率越高，质量略有下降
2. **速度权衡**：压缩需要额外时间，但减少了LLM处理时间
3. **成本节省**：所有压缩方法都能显著降低成本（40-60%）
4. **最佳实践**：ContextualCompression在质量和成本间平衡最好
