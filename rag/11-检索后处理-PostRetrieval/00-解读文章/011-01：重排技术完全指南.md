## 前言

前面我们学习了索引优化技术，知道如何构建高效的索引结构。但即使有了优化的索引，初步检索的结果往往还不够精准。向量相似度分数并不完全等同于真实的相关性，这时候就需要重排技术来进一步优化检索结果。

重排就像图书馆管理员的二次筛选：先快速找出可能相关的100本书，再仔细查看内容，挑选出最相关的3本。这篇文章将介绍6种主流的重排技术，帮助你显著提升RAG系统的检索精度。

## 什么是重排（Reranking）？

### 用生活例子理解重排

想象你在图书馆找书：

第一步：粗筛（初步检索）
- 你告诉图书管理员："我想找关于山西旅游的书"
- 管理员快速给你找了100本可能相关的书
- 这一步很快，但不够精准

第二步：精选（重排）
- 你仔细翻看这100本书的目录和简介
- 根据内容质量重新排序
- 最终选出最相关的3本书
- 这一步慢一些，但非常精准

重排就是第二步！它让检索结果更准确。

## 为什么需要重排？

### 问题场景

假设你问："山西有哪些著名景点？"

只用向量检索（没有重排）：
```
结果1: "山西省位于华北地区..." (相似度 0.85) 太宽泛
结果2: "五台山是佛教名山..." (相似度 0.82) 很相关
结果3: "云冈石窟的历史..." (相似度 0.80) 很相关
```

问题：相似度分数不等于真实相关性！

加上重排后：
```
结果1: "五台山是佛教名山..." (重排分数 0.95)
结果2: "云冈石窟的历史..." (重排分数 0.92)
结果3: "平遥古城是世界遗产..." (重排分数 0.88)
```

重排让真正相关的内容排在前面！

## 技术详解

### 1. RRF重排（倒数排名融合）

**文件：** `01-RRF重排.py`

#### 核心思想

把多个检索结果"投票"融合在一起。

#### 通俗例子

你想买手机，问了3个朋友推荐：
- 朋友A推荐：iPhone > 小米 > 华为
- 朋友B推荐：小米 > iPhone > OPPO  
- 朋友C推荐：华为 > 小米 > iPhone

**RRF算法计算：**
```
iPhone分数 = 1/(60+1) + 1/(60+2) + 1/(60+3) = 0.049
小米分数 = 1/(60+2) + 1/(60+1) + 1/(60+2) = 0.049
华为分数 = 1/(60+3) + 1/(60+3) + 1/(60+1) = 0.048
```

小米和iPhone得分最高！（多次出现在前排）

#### 工作流程

```
用户问题："山西有哪些景点？"
    ↓
生成多个查询：
  - "山西著名旅游景点"
  - "山西必去的地方"  
  - "山西景区推荐"
    ↓
分别检索，得到3个结果列表
    ↓
RRF融合 → 最终排序
```

#### 优势
- 简单高效，无需训练模型
- 提高召回率（不同角度查询）
- 降低单一查询的偏差

#### 劣势
- 需要生成多个查询（调用LLM）
- 精度不如深度模型

#### 适用场景
- 混合检索（BM25 + 向量检索）
- 多语言检索融合
- 快速原型开发

### 2. CrossEncoder重排（交叉编码器）

**文件：** `02-CrossEncoder重排.py`

#### 核心思想

把查询和文档"一起"输入BERT模型，让模型深度理解它们的关系。

#### 通俗例子

**普通方法（双塔模型）：**
```
查询 → 编码器A → 向量1
文档 → 编码器B → 向量2
计算向量1和向量2的相似度
```
就像两个人各自写自我介绍，然后比较文字相似度。

**CrossEncoder方法：**
```
查询 + 文档 → 一起输入BERT → 直接输出相关性分数
```
就像让两个人面对面聊天，判断他们是否合得来。

#### 工作原理

```
输入格式：[CLS] 查询 [SEP] 文档 [SEP]

例子：
[CLS] 山西有哪些景点 [SEP] 五台山是佛教名山 [SEP]
    ↓
BERT深度理解
    ↓
[CLS]位置输出 → 相关性分数：0.95
```

#### 为什么精度高？

CrossEncoder能理解：
- **词序关系**："北京到上海" vs "上海到北京"
- **语义交互**："便宜的好货" vs "便宜没好货"
- **上下文**："苹果手机" vs "苹果水果"

#### 优势
- 精度最高（通常提升10-20%）
- 深度语义理解
- 开源免费

#### 劣势
- 速度慢（每对都要单独计算）
- 不能预计算（查询和文档必须一起编码）
- 只适合重排阶段（不能处理百万级文档）

#### 适用场景
- 对精度要求高的应用
- 重排top 10-100个候选文档
- 有GPU资源的环境

#### 代码示例

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification

# 加载模型
model_name = "cross-encoder/ms-marco-MiniLM-L-12-v2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForSequenceClassification.from_pretrained(model_name)

# 计算相关性
query = "山西有哪些景点？"
doc = "五台山是佛教名山"

inputs = tokenizer(query, doc, return_tensors="pt")
score = model(**inputs).logits[0][0].item()
print(f"相关性分数: {score:.4f}")
```

### 3. ColBERT重排（后期交互）

**文件：** `03-CoBERT重排.py`

#### 核心思想

结合双塔模型的速度和CrossEncoder的精度。

#### 通俗例子

**传统方法的问题：**
- 双塔模型：快但不够准（各自编码，简单比较）
- CrossEncoder：准但太慢（一起编码，无法预计算）

**ColBERT的创新：**
```
第一步：分别编码（可以预计算）
  查询 → BERT → 每个词的向量
  文档 → BERT → 每个词的向量

第二步：精细匹配（MaxSim操作）
  查询的每个词 → 找文档中最相似的词 → 累加分数
```

就像：
- 你说："我想吃好吃的川菜"
- 餐厅A："正宗四川火锅，味道好"
- 餐厅B："粤菜海鲜，新鲜美味"

ColBERT会匹配：
- "川菜" ↔ "四川" ✅ 高分
- "好吃" ↔ "味道好" ✅ 高分
- 餐厅A总分更高！

#### MaxSim操作详解

```python
# 对查询的每个词
for query_token in query_tokens:
    # 找到文档中最相似的词
    max_sim = max(similarity(query_token, doc_token) 
                  for doc_token in doc_tokens)
    total_score += max_sim
```

#### 优势
- 速度快（文档可预编码）
- 精度高（token级别交互）
- 可扩展（支持大规模检索）

#### 劣势
- 需要专门训练的ColBERT模型
- 存储开销大（每个token都有向量）

#### 适用场景
- 大规模文档检索（百万级）
- 需要平衡速度和精度
- 第一阶段检索（不仅是重排）

### 6. 时效加权重排（时间因素）

**文件：** `06-时效加权重排.py`

#### 核心思想

考虑文档的"新鲜度"，越新的文档权重越高。

#### 通俗例子

你搜索"iPhone价格"：
- 文档A：2020年的价格信息（相似度0.9）
- 文档B：2024年的价格信息（相似度0.8）

**普通检索：** 返回文档A（相似度更高）❌  
**时效加权：** 返回文档B（更新鲜）✅

#### 时间衰减公式

```python
最终分数 = 语义分数 × 时间衰减因子

时间衰减因子 = exp(-衰减率 × 经过的小时数)

例子：
- 衰减率 = 0.5
- 文档1小时前访问：衰减因子 = exp(-0.5×1) = 0.61
- 文档24小时前访问：衰减因子 = exp(-0.5×24) = 0.000006
```

#### 衰减率的影响

```
衰减率 = 0.1（衰减慢）
- 1小时后：权重保留90%
- 24小时后：权重保留8%

衰减率 = 0.5（衰减中等）
- 1小时后：权重保留61%
- 24小时后：权重几乎为0

衰减率 = 1.0（衰减快）
- 1小时后：权重保留37%
- 24小时后：权重完全消失
```

#### 使用示例

```python
from langchain.retrievers import TimeWeightedVectorStoreRetriever

retriever = TimeWeightedVectorStoreRetriever(
    vectorstore=vectorstore,
    decay_rate=0.5,  # 衰减率
    k=5              # 返回5个文档
)

# 添加文档时指定时间
from datetime import datetime, timedelta

doc1 = Document(
    page_content="昨天的新闻",
    metadata={"last_accessed_at": datetime.now() - timedelta(days=1)}
)

doc2 = Document(
    page_content="今天的新闻",
    metadata={"last_accessed_at": datetime.now()}
)

retriever.add_documents([doc1, doc2])

# 检索时自动考虑时间因素
results = retriever.get_relevant_documents("新闻")
# doc2会排在前面（更新鲜）
```

#### 优势
- 自动优先返回新内容
- 适合动态更新的知识库
- 实现简单，开销小

#### 劣势
- 需要维护时间元数据
- 可能忽略经典内容
- 衰减率需要调优

#### 适用场景
- 新闻检索系统
- 实时信息查询（股票、天气）
- 知识库维护（优先新文档）
- 社交媒体内容

#### 参数调优建议

```python
# 新闻类应用（快速衰减）
decay_rate = 1.0  # 24小时后基本失效

# 技术文档（慢速衰减）
decay_rate = 0.01  # 几个月后仍有效

# 社交媒体（中等衰减）
decay_rate = 0.5  # 几天后衰减明显
```

## 如何选择重排技术？

### 决策树

```
开始
  ↓
是否需要考虑时间因素？
  ├─ 是 → 时效加权重排
  └─ 否 ↓
       ↓
是否有预算？
  ├─ 有 → Cohere重排（最省心）
  └─ 无 ↓
       ↓
是否需要最高精度？
  ├─ 是 → CrossEncoder（精度最高）
  └─ 否 ↓
       ↓
是否需要大规模检索？
  ├─ 是 → ColBERT（速度+精度平衡）
  └─ 否 ↓
       ↓
是否需要融合多个检索器？
  ├─ 是 → RRF重排
  └─ 否 → CrossEncoder（通用选择）
```

### 场景推荐

#### 场景1：电商搜索
```
需求：快速、准确、大规模
推荐：ColBERT
理由：支持百万级商品，速度快，精度高
```

#### 场景2：企业知识库
```
需求：高精度、中等规模、成本可控
推荐：CrossEncoder
理由：精度高，开源免费，适合企业内部部署
```

#### 场景3：新闻推荐
```
需求：考虑时效性、实时更新
推荐：时效加权重排
理由：自动优先推荐最新内容
```

#### 场景4：快速原型
```
需求：快速上线、效果好、不想维护
推荐：Cohere
理由：开箱即用，效果好，省时省力
```

#### 场景5：法律/医疗检索
```
需求：最高精度、可解释性
推荐：RankLLM
理由：理解能力强，能处理复杂推理
```

## 实战：组合使用多种重排技术

### 多阶段重排Pipeline

```python
# 第一阶段：快速召回（向量检索）
docs = vector_retriever.invoke(query, k=1000)
# 1000个候选文档

# 第二阶段：粗排（RRF融合）
docs = rrf_rerank(query, docs, top_k=100)

# 第三阶段：精排（CrossEncoder）
docs = crossencoder_rerank(query, docs, top_k=20)
# 缩减到20个

# 第四阶段：时效调整（如果需要）
docs = time_weighted_adjust(docs)
# 考虑时间因素

# 最终返回top 5
return docs[:5]
```

### 混合策略示例

```python
def smart_rerank(query, docs, query_type):
    """
    根据查询类型选择重排策略
    """
    if query_type == "news":
        # 新闻类查询：时效性最重要
        return time_weighted_rerank(query, docs)
    
    elif query_type == "factual":
        # 事实类查询：精度最重要
        return crossencoder_rerank(query, docs)
    
    elif query_type == "complex":
        # 复杂查询：需要推理
        return rankllm_rerank(query, docs)
    
    else:
        # 默认：RRF融合
        return rrf_rerank(query, docs)
```

## 重要知识点

### 1. 学习排序（Learning to Rank, LTR）

#### 什么是LTR？

传统重排方法是"固定"的，而LTR可以"学习"如何排序。

#### 三种LTR方法

**Pointwise（逐点法）：**
```
把排序问题变成回归/分类问题
预测每个文档的相关性分数
```

**Pairwise（成对法）：**
```
学习文档对的相对顺序
判断：文档A是否比文档B更相关？
```

**Listwise（列表法）：**
```
直接优化整个排序列表
目标：让整体排序最优
```

#### 常用LTR算法

- **LambdaMART**：基于梯度提升的排序算法
- **RankNet**：神经网络排序
- **ListNet**：列表级别的神经网络

#### 何时使用LTR？

- 有大量标注数据（查询-文档-相关性）
- 需要融合多个特征（文本、点击、时间等）
- 对排序效果要求极高

### 2. 负样本挖掘（Hard Negative Mining）

#### 什么是负样本挖掘？

训练重排模型时，不仅需要正样本（相关文档），还需要"难"的负样本。

#### 为什么重要？

```
简单负样本：
查询："山西景点"
负样本："如何做红烧肉"
→ 太容易区分，模型学不到东西

困难负样本：
查询："山西景点"
负样本："陕西景点推荐"
→ 很相似但不相关，模型需要仔细区分
```

#### 挖掘方法

```python
# 1. 使用初始检索器找到"看起来相关"的文档
candidates = retriever.invoke(query, k=100)

# 2. 过滤掉真正相关的
hard_negatives = [doc for doc in candidates 
                  if doc not in relevant_docs]

# 3. 用于训练
train_triplets = (query, positive_doc, hard_negative)
```

### 3. 蒸馏（Distillation）

#### 什么是模型蒸馏？

让小模型学习大模型的能力。

#### 在重排中的应用

```
教师模型（大）：CrossEncoder-Large
学生模型（小）：CrossEncoder-Mini

步骤：
1. 用大模型对大量查询-文档对打分
2. 用这些分数训练小模型
3. 小模型学会模仿大模型
4. 部署小模型（速度快，效果接近大模型）
```

#### 优势

- 速度提升3-5倍
- 精度只下降2-3%
- 适合生产环境

### 4. 查询扩展（Query Expansion）

#### 什么是查询扩展？

在重排前，先丰富查询内容。

#### 方法

**同义词扩展：**
```
原查询："手机"
扩展后："手机 智能手机 移动电话"
```

**伪相关反馈（PRF）：**
```
1. 用原查询检索top 3文档
2. 提取这些文档的关键词
3. 加入原查询
4. 重新检索
```

**LLM扩展：**
```python
prompt = f"用户查询：{query}\n请生成3个相关的扩展查询"
expanded_queries = llm(prompt)
```

### 5. 多模态重排

#### 什么是多模态重排？

不仅考虑文本，还考虑图片、表格等。

#### 应用场景

```
电商搜索：
查询："红色连衣裙"
文档1：文本描述 + 商品图片
文档2：只有文本描述

多模态重排会优先返回文档1（有图片）
```

#### 实现方法

```python
# 1. 分别编码文本和图片
text_emb = text_encoder(text)
image_emb = image_encoder(image)

# 2. 融合特征
combined_emb = concat([text_emb, image_emb])

# 3. 计算相关性
score = model(query_emb, combined_emb)
```

### 6. 在线学习（Online Learning）

#### 什么是在线学习？

根据用户反馈实时更新重排模型。

#### 工作流程

```
1. 用户搜索"山西景点"
2. 系统返回排序结果
3. 用户点击了第3个结果
4. 系统记录：第3个结果应该排更前
5. 更新模型参数
6. 下次类似查询，第3个结果会排更前
```

#### 常用信号

- **点击**：用户点击了哪个结果
- **停留时间**：在结果页停留多久
- **跳出率**：是否立即返回搜索
- **转化**：是否完成目标动作

### 7. A/B测试

#### 为什么需要A/B测试？

离线指标（MRR、NDCG）不等于用户满意度！

#### 实验设计

```
对照组（A）：使用CrossEncoder重排
实验组（B）：使用Cohere重排

分流：50%用户看A，50%看B

观察指标：
- 点击率
- 转化率
- 用户满意度
- 平均停留时间
```

#### 注意事项

- 样本量要足够大（至少几千用户）
- 实验时间要足够长（至少1-2周）
- 控制其他变量（同一时间段、同一用户群）
